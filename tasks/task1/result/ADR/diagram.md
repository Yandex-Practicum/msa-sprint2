# Контекст и текущие проблемы

Текущее приложение Hotelio реализовано как монолитное Spring Boot-приложение, где модули Booking, User, Hotel, PromoCode и Review находятся в одном коде, разворачиваются как единое приложение и используют одну базу данных PostgreSQL.

Текущая архитектура создаёт следующие проблемы:

- сложность сопровождения: любое изменение требует понимания значительной части кодовой базы;
- высокая связанность: BookingService напрямую зависит от HotelService, UserService, PromoCodeService и ReviewService; В будущем потребуется применение паттерна BFF.
- низкая масштабируемость: нельзя масштабировать только наиболее нагруженный компонент;
- неравномерная производительность: пиковая нагрузка на один сценарий приводит к масштабированию всего приложения;
- невозможность гибкой разработки: разные команды не могут независимо выпускать изменения;
- сложности с быстрым CI/CD: релиз даже небольшой функциональности требует поставки всего приложения;
- высокий риск внедрения новых фич: плотная связанность усложняет тестирование;
- единая база данных выступает общей точкой связанности между доменами.
- отсутствие сервиса логгирования и трассировок;
- рамках анализа были выявлены не только текущие проблемы монолита, но и дополнительные архитектурные ограничения, включая отсутствие слоя адаптации API под разные клиентские платформы. Для решения этой проблемы в целевом состоянии может быть введён BFF/API Gateway. Однако, поскольку по условию задания миграция должна начаться с одного модуля, внедрение BFF не включается в первую итерацию и рассматривается как один из следующих этапов перехода к целевой архитектуре.
# Решение

Принято решение мигрировать Hotelio от монолита к микросервисной архитектуре по паттерну **Strangler Fig**, начиная с выделения одного модуля — **Promo Service**.

## Общая стратегия

На первом этапе:

- монолит остаётся основной системой и сохраняет внешний входной REST API;
- из монолита выделяется только модуль промокодов;
- внутри монолита локальные вызовы PromoCodeService заменяются на вызовы внешнего Promo Service;
- данные промокодов переводятся под владение нового сервиса;
- остальная бизнес-логика пока остаётся внутри монолита.

Таким образом, первый этап соответствует ограничению задания: миграция начинается **с одного модуля**, без изменения всей архитектуры сразу.

## Почему выбран именно Promo Service

Promo Service выбран первым, потому что:

1. у него чёткая и ограниченная бизнес-граница;
2. его данные относительно легко выделить в отдельное хранилище;
3. он связан с ключевым сценарием бронирования, но не является всей его оркестрацией;
4. его выделение позволяет проверить основной механизм Strangler Fig без слишком высокого риска;
5. сбой в промокодах менее критичен, чем сбой в полном флоу бронирования.

## Что меняется на первом этапе

На первом этапе:

- из монолита выносится логика промокодов;
- создаётся отдельный Promo Service;
- создаётся отдельная Promo DB;
- внутри монолита появляется `PromoGatewayClient` для вызова нового сервиса;
- `BookingService` вместо локального вызова `PromoCodeService` обращается к новому сервису удалённо;
- внешний клиентский API может оставаться прежним;
- при необходимости запросы `/api/promos` могут временно проксироваться через монолит.

## Логика принятия решения

Выбор сделан по принципу: **максимально безопасный первый шаг**

Выделение BookingService первым было отклонено, так как это наиболее связанный чувствительный модуль.  
Выделение Promo Service, напротив, позволяет:

- начать декомпозицию;
- обкатать межсервисные вызовы;
- проверить практики отдельного деплоя и отдельной БД;
- внедрить observability и rollback;
- сохранить основной пользовательский флоу внутри монолита.

# План миграции по Strangler Fig

## Этап 0. Подготовка монолита

Перед физическим выделением промокодов необходимо логически изолировать этот модуль внутри монолита:

- выделить единый интерфейс обращения к промокодам;
- убрать прямые вызовы промокодной логики в обход фасада;
- убедиться, что BookingService работает с промокодами только через один слой абстракции.

## Этап 1. Создание Promo Service

Создать новый микросервис Promo Service:

- отдельное Spring Boot-приложение;
- отдельный REST API;
- отдельная PostgreSQL БД;
- отдельная реализация:
  - получения промокода,
  - валидации промокода,
  - проверки применимости,
  - расчёта скидки.

## Этап 2. Выделение данных

Перенести ownership данных промокодов в новый сервис:

- выделить данные `promo_code` в отдельную БД;
- зафиксировать, что только Promo Service владеет этими данными;
- не оставлять общую БД как постоянное решение.

## Этап 3. Интеграция монолита с новым сервисом

В монолите реализовать `PromoGatewayClient`, который вызывает Promo Service.

После этого:

- `BookingService` использует внешний сервис для проверки промокодов;
- клиентский API для бронирования остаётся неизменным;
- остальная логика бронирования по-прежнему живёт в монолите.

## Этап 4. Feature flag и rollback

Добавить конфигурационный переключатель:

- локальная реализация;
- удалённая реализация.

Это позволит:

- безопасно включать новую логику;
- быстро откатываться;
- сравнивать поведение старой и новой реализации.

## Этап 5. Наблюдаемость и стабилизация

Добавить:

- correlation id / trace id;
- технические метрики;
- бизнес-метрики;
- monitoring latency/error rate;
- алерты на отказ Promo Service.

## Этап 6. Удаление старой реализации из монолита

После стабилизации:

- удалить старый PromoCodeService из монолита;
- оставить только интеграцию с внешним Promo Service;
- окончательно вывести промокоды из ответственности монолита.

## Этап 7. Следующие шаги миграции

После успешного выделения Promo Service можно планировать следующий модуль для декомпозиции.

Наиболее вероятные кандидаты:
- Review Service;
- Hotel Service.

BookingService рекомендуется оставлять на более поздний этап, когда уже будут отработаны практики интеграции, наблюдаемости и владения данными.

# Решение по диаграммам C4

В рамках ADR принята следующая логика моделирования:

- **C1 Context** показывает переходное состояние системы: пользователь работает с монолитом, а монолит вызывает новый Promo Service;
- **C2 Container** показывает монолит, отдельный Promo Service, их БД и систему наблюдаемости;
- **C3 Component** показывает внутреннюю перестройку монолита: удаление внутреннего PromoCodeService как доменного модуля и замену его на `PromoGatewayClient`.

# Альтернативы

## Альтернатива 1. Выделить первым Booking Service

### Плюсы
- ранняя декомпозиция core-domain;
- теоретически сильный архитектурный эффект.

### Недостатки
- максимальная связанность с другими модулями;
- высокий риск поломки основного пользовательского сценария;
- необходимость решать слишком много архитектурных проблем сразу;
- сложный rollback.

## Альтернатива 2. Выделить первым Hotel Service

### Плюсы
- полезен для сценариев чтения и масштабирования каталога;
- имеет понятную предметную область.

### Недостатки
- сильнее затрагивает клиентские сценарии поиска и отображения;
- менее удобен как первый шаг, чем промокоды;
- может потребовать более раннего изменения клиентских контрактов.

## Альтернатива 3. Выделить первым Review Service

### Плюсы
- ограниченный домен;
- низкий риск.

### Недостатки
- меньшее влияние на основной flow бронирования;
- меньше практической пользы для проверки интеграции в критичном сценарии.

## Альтернатива 4. Ввести BFF на первом этапе

### Плюсы
- улучшение структуры входного слоя;
- подготовка API под разные фронтенды.

### Недостатки
- выходит за рамки ограничения “начать с одного модуля”;
- не является доменным модулем;
- увеличивает объём изменений и сложность первой итерации.

## Альтернатива 5. Выполнить Big Bang migration

### Плюсы
- быстрое достижение целевой архитектуры только в теории.

### Недостатки
- крайне высокий риск;
- большой объём изменений;
- сложный rollback;
- высокая вероятность длительной нестабильности системы.

# Недостатки, ограничения, риски выбранного решения

1. После появления первого микросервиса возрастает эксплуатационная сложность.
2. Локальные вызовы заменяются сетевыми вызовами, что увеличивает latency.
3. Появляется риск недоступности Promo Service.
4. На переходном этапе архитектура становится смешанной и временно более сложной.
5. Feature flag может привести к временному дублированию логики.
6. Появляется необходимость контрактных и интеграционных тестов.
7. Команда должна поддерживать одновременно монолит и новый сервис.

# Меры снижения рисков

- использовать timeout, retry и circuit breaker;
- предусмотреть controlled fallback при недоступности Promo Service;
- использовать feature flag для безопасного переключения;
- внедрить централизованные логи, метрики и трассировку;
- покрыть интеграцию контрактными тестами;
- выполнять миграцию по одному bounded context за раз.

# Итог

Для Hotelio рекомендуется поэтапная миграция от монолита к микросервисной архитектуре по паттерну Strangler Fig.  
Первым шагом следует выделить один модуль — Promo Service — без введения BFF на этом этапе. Такой подход соответствует ограничению задания, снижает риск миграции, позволяет обкатать базовые практики микросервисной архитектуры и создаёт основу для дальнейшей декомпозиции системы.
