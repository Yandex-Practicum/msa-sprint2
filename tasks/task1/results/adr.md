### **Целевое решение**
В файле [tobe.puml](tobe.puml) приведена C2 схема целевого разделения монолита на микросервисы
Разбивать монолит на микросервисы было решено по DDD, т.к. переход на микросервисы вызван требованием соответсвовать быстрорастущему бизнесу. Необходима более гибкая архитектура для реализации бизнес-фич. 

В соответстивии с этим были выделены основные бизнес-домены монолита:
- пользователи (users service) - содержит логику по работе с пользователями, хранит данные об их состоянии и статусах. 
- бронирования (booking service) - содержит логику оформления бронирований и их хранение
- отзывы и рейтинги отелей (review service) - содержит логику по отзывам об отелях, их рейтингу, обеспечивает их хранение
- отели (hotel service) - предоставляет информацию об отелях. Используется для валидации бронирования и отображения информации в интерфейсе.
- промокоды (promoCode service) - логика работы с промокодами и их хранение

### **Как целевое решение исправит проблемы монолита**
#### Cложность сопровождения
**Проблема:** любое изменение требует понимания всей кодовой базы и невозможно менять один модуль, не затрагивая другие <br>
  **Решение:** т.к. доменные зоны разделены на отдельные микросервисы, то внесение изменений будет происходить в отдельно взятом репозитории/проекте. Даже при реализации крупной бизнес-фичи, затрагивающей множество микросервисов, изменения будут производится в разделенных проектах, что позволит реализовывать фичу поэтапно, разными командами параллельно     
####  Низкая масштабируемость
**Проблема:** при пиковых нагрузках нельзя масштабировать только нужный компонент; производительность неравномерна<br>
**Решение:** разделение на микросервисы даст возможнсть масштабировать компонетны системы независимо друг от друга и в требуемое время (применение k8s и переезд в cloud в перспективе). Применение механизмов асинхронного взаимодействия (kafka), позволит смягчить уровень нагрузки на систему в операциях, не требующих моментального результата (например, процесс бронирования, создания отзывов и рейтинга)
#### Невозможность гибкой разработки
**Проблема:** разные команды не могут работать независимо; трудно внедрить быстрый CI/CD<br>
**Решение:** микросервисы разделены на домены, что позволит развивать бизнес-кусочки системы с минимизацией зависимостей от других сервисов / команд. Проблема внедрения CI/CD так же разрешиться, т.к. внесения изменений в одни сервисы не будет афектить другие сервисы, а командам не потребуется синхронизации перед релизом.

####  Ограничения на фронтенде
**Проблема:** REST API монолита слишком универсален и плохо адаптируется под разные платформы нет поддержки BFF<br>
**Решение:** Используем 2 BFF: REST API и Apollo Federation GraphQL. Каждый микросервис будет реализовывать свой собственный GraphQL и REST API протокол для своей предметной области. (тут честно не знаю, мб стоит просто использовать Hasura GraphQL Engine и подключить ему бд микросервисов, так минимизируем кол-во доработок, но как будто теряется изолированность компонентов). Таким образом даем гибкость и инкапсуляцию системы для клиента.
Между сервисами оставляем взаимодействие по REST API, т.к. оно уже там есть.


#### Сложности с запуском новых фич
**Проблема:** большой риск ошибок из-за плотной связанности; тестирование требует понимания всех зависимостей <br>
**Решение:** изолированность сервисов при микросервисной архитектуре уменьшает связность и увеличивает гибкость тестирования отдельных компонентов и всей системы в целом 

### План поэтапного перехода на микросервисы
Для поэтапного перехода требуется определиться с ранжированием выделяемых компонентов, учиывая факторы бизнес-целей, технических возможностей и интересов клиентов.
Из описания к задаче выделил следующие факторы, на которые стоит сделать акцент при ранижировании:
- у Hotelio около 1 млн зарегистрированных пользователей и основной трафик приходится на поиск и бронирование жилья. 
- в проблемах указано в качестве примера, что пиковые нагрузки приходятся на бронирования

Выводы:
Ранжирование с точки зрения бизнес-влияния: сервисы "бронирования", "отели"
- сервис "бронирования", "отели" - являются приоритетными для растущего бизнеса т.к. на них лежит его ядро. Т.к. нагрузка на систему растет, то ограничения монолита в данный момент могут повлечь за собой репутационные издержки, недополучение потенциальной прибыли из-за ухода недовольных клиентов, потеря конкурентного преимущества. Например, медленная работа системы в пиковые периоды, скудный функционал, медленное внедрение бизнес-фич может вынудить пользователей оставлять плохие отзывы в интернете, уход к конкурентам.  

Ранжирование с точки зрения технического влияния: сервисы "промокоды" и "отзывы и рейтинги отелей"
- с технической точки зрения лучше начинать с простых и малосвязных компонентов. Это позволит командам постепенно адаптироваться к микросервисной архитектуре, заложить технический фундамент: настроить CI/CD для prod и staging окружений, выработать подход к тестированию, изучить новые технологии и опробовать их на компонентах системы, чей отказ не приведет к отказу всей системы (но может навредить репутации)

Ранжирование с точки зрения клиентского влияния: сервисы "бронирования", "отели", "отзывы и рейтинги"
- сложно сказать, т.к. не приведена обратная связь от пользователей. Исходя из того, что основной трафик приходится на поиск и бронирование жилья, косвенно можно сделать вывод, что пользователям важно иметь быстрый и качественный доступ к основным компонентам системы: "бронирования", "отели", "отзывы и рейтинги". Первые три системы обеспечивают основной функционал, последний - кмк важен для пользователя, т.к. именно "отзывы и рейтинги" играют ключевую роль при выборе отеля.

Итого:
Сервисы "бронирования", "отели" - наиболее приоритетны.
Сервисы  "пользователи", "промокоды" и "отзывы и рейтинги отелей" - менее приоритетны.

Для приоритезации выберем метод MoSCoW (т.к. проект имеет максимально тесное влияние на бизнес):

**Обязательно**:
- "бронирования" - прямой источник дохода бизнеса, самый ключевой функционал для жизни системы. При этом самый нагруженный
- "отели" - основной контент системы + поиск. 

**Нужно**:
- "отзывы и рейтинги отелей" - имеет чуть ниже значимость чем "отели" и "бронирования" для клиента. Но при растущем потоке клиентов, функциональность может начать страдать и давать негативный опыт клиенту. Поэтому этим компонентом стоит заняться во 2ю очередь. Возможно, в качестве временной поддержки компонента стоит рассмотреть применение механизмы асинхронной работы и кеширования. Например, создание отзывов и изменение пользователями рейтинга отелей может быть перенесено в асинхронный режим, а сами отзывы и рейтинги можно кешировать

**Желательно**
- "промокоды" - кмк стоит брать следующим: промокоды применяются далеко не к каждому заказу, а массовый выпуск новых может быть временно заморожен до переезда. Отнес в категорию "Нужно" т.к имеет прямое влияние на клиента, но не существенное и не частое. Функционал на короткой перспективе кмк не имеет мест для расширения

**Можно перенести**
- "пользователи" - расширяемость данного модуля с бизнес точки зрения слабо представляется. Разнообразие хранимых данных о пользователях не велико (id, статус, имя, email итд) и кажется изменится не существенно в ближайщей перспективе. 1 млн записей (даже с перспективой роста в 2-3 раза) для Postgres не проблема, при этом данные редко меняются и могут быть закешированы при необходимости. Вообще, после выезда всех остальных компонентов системы, этот функционал можно никуда и не выносить т.к. он останется единственным, что по сути, сделает из него микросервис автоматически.


#### Этап 1 
1) Внедрение API-Gateway. Стоит начать с него т.к. с помощью него можно будет:
- сделать гибкую прослойку между клиентами (web/mibile app), чтобы изолировать их от деталей системы, переезда и выбора "куда ходить" за той или иной информацией
- маршрутизация трафика позволит реализовать канареечные релизы при включении микросервиса в работу, что позволит перенаправлять только часть пользователей на выводимый в эксплуатацию микросервис
- реализовать мониторинг и аналитику над входящим трафиком для отслеживания работы текущей и будущей системы.
- вынести функционал авторизации и аутентификации пользователей
- реализовать кеш данных
3) (Observability) Организация систем мониторинга, алертинга, логирования. Что обеспечит прозрачность как при переезде так и после него
4) настроить в api-gateway все API схемы и направить их на монолит
5) настроить Observability для api-gateway
5) переключать клиентов на api-gateway поэтапно, с возможностью вернуться на хождение в монолит напрямую (выпустить апдейт с маршрутизацией на клиенте)

#### Этап 2 
Выносим сервис "бронирования": 
- создаем отдельный микросервис
- даем доступ сервису до монолитной бд
- реализуем API по получению бронирований: /api/bookings и /api/bookings?userId=[id] (т.к самые простые)
- переключаем трафик c помощью api-gateway, мониторим, в случае чего переключаем трафик обратно на монолит
- реализуем POST /bookings: для него понадобиться: 
  - валидация клиента и расчет basePrice: используем ручку в монолите GET /api/users/{userId} для получения инфы по пользователю
  - валидация отеля: ручки в монолите /api/hotels/{id}/operational, /api/hotels/{id}/fully-booked, /api/reviews/hotel/{hotelId}/trusted
  - валидация промо: ручки в монолите  POST /api/promos/validate
  - после нужно создать заказ - сохранить в бд. Сохраняем в монолите
  - плавно переключаем трафик создания бронирований на микросервис
- заводим отдельную бд для микросервиса
  - начинаем сохранять бронирования в новую бд
  - механизм получения списка бронирований /api/bookings меняем так чтобы делал поиск и по новой бд и по старой + мерж результатов
  - фоном копируем бронирования из старой бд в новую, после чего список бронирований отдает только из новой бд.

Итог: 
 компонент по работе с бронирования выехал из монолита в микросервис, имеет свою бд, 100% трафика по бронированиям на нем. Для подстраховки мб стоило бы делать запись и новую и в старую бд, дабы в случае отказа микросервиса можно было бы переключить трафик с него на монолит, но пока что выглядит избыточно

Альтернативное решение:
 В приниципе оформлять бронирования можно и асинхронно: при запросе на создание брони сервер мог бы генерировать задачу на создание бронирования, сохранять ее в бд и класть в очередь на обработку, а клиенту отдавать id задачи. Сам клиент мог бы через long polling или SSE опрашивать бек с этим id. Плюс подхода - масштабируем самую нагруженную часть системы, минусы - требует доработок и на клиентах и на бекенде + внедрение очередей, что затянет переезд. Можно рассмотреть на последних этапах

#### Этап внедрения BFF-ов
