#### Обоснование архитектуры и технологий:

* **Разделение по бизнес-доменам:** Повышает независимость команд разработки, упрощает понимание и изменение отдельных частей системы, уменьшает вероятность конфликтов при разработке.
* **API Gateway:** Обеспечивает единый вход в систему, позволяет централизованно управлять аутентификацией, авторизацией, логированием, кешированием, безопасностью и политиками доступа.
* **Собственные БД для сервисов:** Поддерживает изоляцию данных и бизнес-логики, позволяет каждому сервису выбирать наиболее подходящую СУБД, упрощает масштабирование.
* **Стратегия Strangler Fig:** Позволяет постепенно и безопасно заменить монолит на микросервисы, минимизируя риски и обеспечивая непрерывность работы.

### **Альтернативы**

Миграция **Review-сервиса** как первого кандидата

* **Преимущества:** Слабые зависимости (в основном от Booking), низкая сложность, небольшой объем данных. Хорош для отработки процесса миграции.
* **Риск:** Меньшая бизнес-ценность по сравнению с Booking.
 
#### Недостатки, ограничения, риски:

* **Сложность миграции:** Требует тщательного планирования, настройки новой инфраструктуры, CI/CD, мониторинга и управления конфигурациями.
* **Сетевые задержки:** Взаимодействие между сервисами по сети может замедлить общую работу по сравнению с внутренними вызовами в монолите.
* **Управление распределёнными транзакциями:** Обеспечение согласованности данных между сервисами сложнее, чем в монолите с одной БД.
* **Опасность полной переписки:** Риск "монолита в облаке" или создания "мега-сервиса" при неправильной декомпозиции.

### **План миграции (Strangler Fig)**

1. Создание API Gateway: Установка и настройка маршрутизации (Spring Cloud Gateway, Consul).
2. Миграция Booking сервиса:
   * Выделение `BookingController`, `BookingService`, `BookingRepository`, сущностей и конфигураций.
   * Создание нового Spring Boot приложения.
   * Настройка отдельной `PostgreSQ`L базы данных.
   * Интеграция с API Gateway (маршрутизация `/api/bookings/**`).
   * Проверка и тестирование нового сервиса.
   * Постепенное отключение старого кода бронирования в монолите.
3. Повторение для других сервисов: `Review`, `PromoCode`, `Hotel`, `AppUser` по аналогии, учитывая зависимости и приоритеты.

### **Выбор первого модуля для миграции: Booking**

**Причины выбора:**

* **Ключевой домен:** Центральный элемент бизнес-процесса системы бронирования.
* **Четкие границы:** Относительно изолированная функциональность с понятными входами и выходами.
* **Высокая ценность:** Улучшение этого сервиса напрямую влияет на пользовательский опыт и бизнес.
* **Потенциал для роста:** Часто требует масштабирования и доработок.
