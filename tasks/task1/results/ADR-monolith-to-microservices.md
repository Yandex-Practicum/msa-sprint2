### **Функциональные требования**
Определяющие пользовательские сценарии и внешние контракты (REST), которые необходимо сохранить без изменений на границе платформы.

| **№** | **Действующие лица или системы** | **Use Case** | **Описание** |
| :-: | :- | :- | :- |
| 1 | Клиент → API | Создание бронирования | POST /api/bookings?userId=&hotelId=&promoCode=*: проверка пользователя, отеля, отзывов, промокода; расчёт цены/скидки; сохранение. |
| 2 | Клиент → API | Список бронирований | GET /api/bookings[?userId=]: возврат всех или пользователя. |
| 3 | Клиент → API | Поиск отелей по городу | GET /api/hotels/by-city?city=*. |
| 4 | Клиент → API | Топ отелей | GET /api/hotels/top-rated?city=*&limit=*. |
| 5 | Клиент → API | Получить промокод | GET /api/promos/{code}. |
| 6 | Клиент → API | Проверка промокода для пользователя | POST /api/promos/validate?code=&userId=. |
| 7 | Клиент → API | Отзывы по отелю | GET /api/reviews/hotel/{hotelId}. |
| 8 | Клиент → API | Доверие к отелю | GET /api/reviews/hotel/{hotelId}/trusted. |
| 9 | Клиент/Партнёры → API | Пользовательские статусы | GET /api/users/{id}/(status|active|blacklisted|authorized|vip). |

### **Нефункциональные требования**
Архитектурно значимые свойства для первого этапа миграции.

| **№** | **Требование** |
| :-: | :- |
| N1 | Обратная совместимость внешних API (/api/*) на границе (без изменений контрактов). |
| N2 | Доступность для сценариев бронирования ≥ 99.9%. |
| N3 | Производительность: p95 ≤ 300 мс для POST /api/bookings при номинальном RPS. |
| N4 | Масштабируемость «горячего пути» (создание брони) независимо от прочих модулей. |
| N5 | Наблюдаемость: метрики, логи, распределённые трассировки по запросу. |
| N6 | Изоляция данных: «один сервис - одна БД». |
| N7 | Предсказуемая обработка ошибок: бизнес-ошибки → 4xx (не 500). |
| N8 | Эволюционность: миграция без даунтайма (паттерн Strangler Fig). |

### **Решение**
- Целевая архитектура: микросервисы Booking, Users, Hotels, Promos, Reviews; каждый владеет своей БД. Синхронные REST-вызовы на первом этапе; далее - события Kafka (например, BookingCreated) для интеграций и асинхронных реакций.
- Внешний контракт (/api/*) сохраняется: вводится фасад (API-шлюз/edge), перенаправляющий трафик к монолиту или выделенным сервисам по мере миграции.
- Обработка ошибок приводится к 4xx для бизнес-сценариев через ControllerAdvice (наследуется в новом сервисе).
- Наблюдаемость: единые трейс-идентификаторы, сбор метрик (RPS/latency/error rate), логи.
- Границы владения данными:
  - Booking → таблица booking;
  - Users → app_user; Hotels → hotel; Promos → promo_code; Reviews → review.
- Диаграмма (C4, контейнеры): tasks/task1/results/to-be-architecture.puml.

Логика выбора технологий и подходов
- Spring Boot для сервисов: соответствие текущему стеку, снижен барьер миграции.
- PostgreSQL per service: явная изоляция данных и схем.
- HTTP/REST сначала - минимальная сложность; Kafka позже - для событийных интеграций.
- API Gateway/BFF в перспективе - для унификации клиентских интеграций (IMPORTANT.md).

### **Альтернативы**
1) Оставить монолит и оптимизировать (кэш, тюнинг БД, вертикальное масштабирование). 
   - Плюсы: низкие начальные затраты; мин. риска изменений.
   - Минусы: не решает независимое масштабирование и скорость релизов.
2) Модульный монолит (жёсткие границы, отдельные модули) как долгосрочное состояние.
   - Плюсы: проще поддерживать, меньше инфраструктурной сложности.
   - Минусы: не даёт операционной изоляции и автономных релизов.
3) «Big Bang» перепись.
   - Плюсы: чистый целевой дизайн.
   - Минусы: высокий риск, длинный «time-to-value», сложная замена в проде.

**Недостатки, ограничения, риски**
- Рост латентности из‑за сетевых вызовов между сервисами - таймауты, ретраи, circuit breaker, кэширование.
- Миграция данных (вынос booking из общей схемы) - требует планирования, временной синхронизации и обратимых шагов.
- Усложнение эксплуатации: логирование, мониторинг, деплой - требуется стандартизовать шаблоны сервисов и CI/CD.
- Согласованность данных: при разбиении границ обеспечить корректную семантику (read-your-writes, eventual для агрегатов).
- Управление контрактами: контрактные тесты, неизменность публичных API на границе.
- Текущая проблема монолита: бизнес-ошибки маппятся в 500 - исправить до начала выноса (ControllerAdvice → 4xx).
