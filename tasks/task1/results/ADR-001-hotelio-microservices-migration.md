# ADR-001: Миграция Hotelio от монолита к микросервисной архитектуре

## Статус
ПРИНЯТО

## Контекст
Система Hotelio в настоящее время реализована как монолитное приложение на Spring Boot с единой PostgreSQL базой данных, обслуживающее ~1млн пользователей. Система сталкивается с серьезными архитектурными проблемами, которые препятствуют масштабированию и развитию:

### Текущие проблемы:
1. **Плотная связанность**: BookingService зависит от всех других сервисов (User, Hotel, PromoCode, Review)
2. **Сложность сопровождения**: любые изменения требуют понимания всей кодовой базы
3. **Низкая масштабируемость**: невозможно масштабировать компоненты независимо
4. **Невозможность гибкой разработки**: команды не могут работать независимо
5. **Ограничения CI/CD**: риск при внесении изменений высок
6. **Единая точка отказа**: падение одного компонента затрагивает всю систему

### Анализ зависимостей:
- **BookingService** — центральный компонент, вызывает все остальные сервисы
- **UserService** — независимый, только читает данные пользователей
- **HotelService** — независимый, управляет каталогом отелей
- **PromoCodeService** — зависит от UserService для проверки VIP статуса
- **ReviewService** — независимый, агрегирует отзывы по отелям

## Решение
Поэтапная миграция к микросервисной архитектуре с применением паттерна **Strangler Fig**.

### Целевая архитектура (через год):
- Независимые микросервисы с собственными базами данных
- Service mesh с автоматизированными rollouts
- Масштабируемая Kafka-инфраструктура для событийного взаимодействия
- Метрики и трассировка на каждый сервис
- GraphQL для фронтенда с BFF (Backend for Frontend)

### Промежуточное состояние (через 2-3 месяца):
- Мигрировано 2-3 сервиса
- Разделение баз данных начато
- Событийная архитектура внедрена частично
- API Gateway развернут

## Выбор первого модуля для миграции: UserService

### Обоснование:
1. **Минимальные зависимости**: не зависит от других сервисов
2. **Четкие границы**: управляет только пользовательскими данными
3. **Стабильный API**: хорошо определенные REST-эндпоинты
4. **Низкий риск**: не влияет критично на бизнес-процессы
5. **Простая модель данных**: только таблица `app_user`
6. **Опыт команды**: позволяет отработать процессы миграции

### Аргументы против других сервисов:
- **BookingService**: слишком сложен, зависит от всех остальных
- **HotelService**: может потребовать интеграции с внешними системами
- **PromoCodeService**: зависит от UserService
- **ReviewService**: может иметь высокую нагрузку при миграции

## Альтернативы
1. **Полная перепишивание** — отклонена из-за высокого риска и времени
2. **Миграция всех сервисов одновременно** — отклонена из-за сложности
3. **Начинать с BookingService** — отклонена из-за сложности зависимостей

## Стратегия миграции (Strangler Fig)

### Фаза 1: Подготовка
1. Создание отдельной базы данных для UserService
2. Репликация данных пользователей
3. Создание UserService как микросервиса
4. Развертывание API Gateway

### Фаза 2: Трафик-переключение
1. Проксирование части запросов на новый UserService
2. A/B тестирование между монолитом и микросервисом
3. Мониторинг производительности и ошибок

### Фаза 3: Полное переключение
1. Перенаправление всего трафика на UserService
2. Удаление UserService из монолита (сохранение как fallback)
3. Очистка неиспользуемого кода

### Фаза 4: Следующие сервисы (ongoing)
1. PromoCodeService (зависит от UserService)
2. ReviewService (независимый)
3. HotelService (независимый)
4. BookingService (последний, самый сложный)

## План технической реализации

### UserService микросервис:
```
- Spring Boot приложение
- Собственная PostgreSQL база
- REST API с теми же эндпоинтами
- Kafka для событий (user created/updated/deleted)
- Docker контейнер
- Health checks и метрики
```

### API Gateway:
```
- Kong или Spring Cloud Gateway
- Маршрутизация запросов
- Аутентификация и авторизация
- Rate limiting
- Мониторинг трафика
```

### Мониторинг:
```
- Prometheus для метрик
- Jaeger для трассировки
- ELK/EFK для логирования
- Grafana для визуализации
```

## Последствия

### Позитивные:
- Команды смогут работать независимо
- Возможность масштабирования отдельных компонентов
- Снижение рисков при разработке
- Улучшение отказоустойчивости
- Возможность использования разных технологий

### Негативные:
- Увеличение сложности инфраструктуры
- Необходимость управления распределенными транзакциями
- Сетевые задержки между сервисами
- Сложность debugging в распределенной системе
- Необходимость мониторинга множества сервисов

## Метрики успеха
- Время развертывания изменений: <30 минут (vs текущие 2+ часа)
- Независимость команд: каждая команда может деплоить без влияния на другие
- Время восстановления после инцидентов: <15 минут
- Масштабируемость: возможность масштабировать отдельные компоненты
- Производительность: сохранение или улучшение времени ответа API
