#Контекст и проблемы текущей архитектуры
##Текущее состояние:

- Монолитное Spring Boot приложение с единой базой данных PostgreSQL
- 5 бизнес-доменов: бронирования, отели, пользователи, промокоды, отзывы
- Высокая связность между сервисами внутри монолита
- Единая точка отказа
- Сложность независимого развертывания компонентов
- Затрудненное масштабирование отдельных функций

##Выявленные проблемы:

- Сложность внесения изменений - изменение в одном модуле может повлиять на другие
- Проблемы с масштабированием - невозможно масштабировать только нагруженные компоненты
- Единая база данных - создает сильную связность между доменами
- Сложность внедрения новых технологий - вся система должна мигрировать одновременно

##Анализ зависимостей модулей:

1. UserService - самый независимый
2. HotelService - часто используется, но сложнее
3. PromoCodeService - простая бизнес-логика
4. ReviewService - зависит от HotelService
5. BookingService - самый сложный, зависит от всех

##Вывод: 

- Предлагается реализовать постепенное разделение монолита на микросервисы, начиная с самого независимого, простого модуля (UserService).
    - Создание нового UserService с собственной БД
    - Реализация двойной записи данных (новый сервис + монолит)
    - Синхронизация данных через события (Kafka)
    - Перенаправление запросов через Feature Flag
