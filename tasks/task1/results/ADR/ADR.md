### ADR-001: Выделение сервиса отзывов из монолита Hotelio по паттерну Strangler Fig

**Статус:** в процессе  
**Дата:** 2026-02-23  
**Контекст системы:** Hotelio (монолит + выделяемый микросервис)

### Контекст

Система Hotelio представляет собой монолитное приложение на Spring Boot, предоставляющее REST API для:

- бронирований (`/api/bookings`)
    
- отелей (`/api/hotels`)
    
- промокодов (`/api/promos`)
    
- пользователей (`/api/users`)
    
- отзывов (`/api/reviews`)
    

Монолит использует общую PostgreSQL БД. Требуется начать постепенную декомпозицию монолита на микросервисы с минимальными рисками, используя паттерн **Strangler Fig** (поэтапное замещение функциональности через маршрутизацию).

### Решение

Первым выделяемым микросервисом выбирается **Reviews Service** (отзывы), обслуживающий эндпоинты `/api/reviews` и имеющий собственную БД (**Review DB**).

Входящий трафик клиентов направляется через **API Gateway (Nginx)**:

- запросы к `/api/reviews` маршрутизируются в `Reviews Service`
    
- остальные запросы маршрутизируются в монолит
    

Монолит в рамках переходного периода может обращаться к `Reviews Service` по сети (например, для чтения данных/агрегатов), не имея прямого доступа к базе `Review DB`.

### Обоснование выбора

1. **Низкий риск для критичных бизнес-транзакций**  
    В отличие от бронирований и промокодов, отзывы не являются частью транзакционного ядра (оплата, создание брони, изменение статусов). Ошибка или деградация функциональности отзывов не должна блокировать возможность бронирования.
    
2. **Чёткая граница домена (bounded context)**  
    Отзывы — отдельная предметная область (оценки, текст, модерация/статусы, агрегаты рейтинга), которую проще изолировать функционально и по данным.
    
3. **Удобная точка strangling через gateway**  
    Путь `/api/reviews` является самостоятельным набором ручек, что позволяет вынести его маршрутизацией без изменений со стороны клиентов.
    
4. **Масштабирование и независимые релизы**  
    Нагрузка на чтение отзывов/рейтингов часто отличается от нагрузки на бронирования. Выделение позволяет масштабировать и выпускать изменения независимо, снижая давление на релизный цикл монолита.
    
5. **Управляемая миграция данных**  
    Отзывы проще переносить и валидировать (backfill + выборочная сверка). Нет необходимости решать проблему распределённых транзакций на первом шаге.
    

### Рассмотренные альтернативы

- **Booking Service** — отклонён как первый кандидат из-за сильной связности с доменами (users/hotels/promos), транзакционной сложности и повышенного бизнес-риска.
    
- **PromoCode Service** — отклонён как первый кандидат, т.к. ошибки могут напрямую влиять на стоимость и выручку.
    
- **User Service** — отклонён как первый кандидат, т.к. является центральной зависимостью для большинства потоков.
    
- **Hotel Service** — возможен, но часто агрегирует данные из разных доменов (включая отзывы/рейтинги), поэтому логичнее выделять после/параллельно с отзывами.
    

### Архитектурный результат

- Добавляется `Reviews Service` (Spring Boot) + `Review DB` (PostgreSQL)
    
- Добавляется/используется `API Gateway` (Nginx) для роутинга по путям
    
- Монолит продолжает обслуживать остальные домены
    
- Интеграция монолита с новым сервисом — через HTTP (при необходимости)
    

### План внедрения (по шагам)

**Этап 1. Подготовка маршрутизации**

- Настроить Nginx/API Gateway: маршрутизация `/api/reviews` → Reviews Service, остальное → монолит
    
- Включить корреляцию запросов (trace-id / request-id) в заголовках между gateway ↔ сервисами
    

**Этап 2. Реализация Reviews Service**

- Поднять новый сервис с ручками `/api/reviews` (как минимум: GET)
    
- Подключить собственную БД и миграции схемы (Flyway/Liquibase)
    

**Этап 3. Миграция данных**

- Выполнить перенос исторических отзывов из монолитной БД в Review DB
    
- Провести сверку: количество записей, выборочные проверки консистентности
    

**Этап 4. Переключение записи**

- Перевести POST/PUT/DELETE по `/api/reviews` на Reviews Service
    
- Ограничить/отключить соответствующие обработчики в монолите
    

**Этап 5. Завершение strangling**

- Удалить/законсервировать код обработки отзывов в монолите
    
- При необходимости — оставить read-only совместимость на ограниченное время
    

### Последствия

**Плюсы**

- Первое уменьшение монолита без риска для транзакционного ядра
    
- Независимые релизы и масштабирование для домена отзывов
    
- Отработка процесса выделения сервиса (gateway, observability, деплой, миграция данных)
    

**Минусы / Риски**

- Появляется межсервисное взаимодействие и требования к устойчивости (таймауты, ретраи)
    
- Возможна eventual consistency для агрегатов рейтинга
    
- Нужны мониторинг и алерты на gateway и новый сервис
    

### Нефункциональные требования

- Логи + метрики + трассировка запросов между gateway/монолитом/Reviews Service
    
- Таймауты и ограничение ретраев на сетевых вызовах
    
- Обратная совместимость API `/api/reviews` для клиентов на период миграции
