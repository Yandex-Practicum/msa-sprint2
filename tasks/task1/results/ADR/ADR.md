### <a name="_b7urdng99y53"></a>**Название задачи:** 
Переход  монолитной цифровой платформы на микросервисную архитектуру
### <a name="_hjk0fkfyohdk"></a>**Автор:**
Бессолицын Антон
### <a name="_uanumrh8zrui"></a>**Дата:**

Сервис Hotelio реализован как единое приложение, в котором все бизнес-функции собраны в одном коде, разворачиваются как единый сервис и используют одну базу данных.

С ростом бизнеса возникли серьёзные архитектурные и операционные проблемы. Команда приняла решение о переходе на микросервисную архитектуру с постепенным выносом сервисов.

Текущие проблемы:
1. Сложность сопровождения
- Любое изменение требует понимания всей кодовой базы.
- Невозможно менять один модуль, не затрагивая другие.
2. Низкая масштабируемость
- При пиковых нагрузках, например на бронирование, нельзя масштабировать только нужный компонент.
- Производительность неравномерна.
3. Невозможность гибкой разработки
- Разные команды не могут работать независимо.
- Трудно внедрить быстрый CI/CD.
4. Ограничения на фронтенде
- REST API монолита слишком универсален и плохо адаптируется под разные платформы.
- Нет поддержки BFF.
5. Сложности с запуском новых фичей
- Большой риск ошибок из-за плотной связанности.
- Тестирование требует понимания всех зависимостей.

Текущая архитектура [схема](./../../../../architecture/C4dia.puml)

Основное функциональное требование - это сохранение всех возможностей монолитной платформы.

**Анализ монолита**

Приложение разделено на несколько модулей
| Модуль             | Назначение |
|--------------------|-----------|
| `BookingService`   | Обрабатывает создание бронирований. Выполняет все проверки: пользователя, отеля, промокода, отзывов. Вычисляет финальную цену, скидку, и сохраняет результат. |
| `UserService`      | Проверяет статус пользователя: активен ли он, не находится ли в чёрном списке, и возвращает его статус. |
| `HotelService`     | Предоставляет информацию об отелях. Используется для валидации бронирования и отображения информации в интерфейсе. |
| `PromoCodeService` | Проверяет промокоды: действительность, применимость к конкретному пользователю. |
| `ReviewService`    | Отвечает за отзывы и рейтинг отеля. Влияет на возможность бронирования, если у отеля плохая репутация. |

Все модули являются внутренними компонентами одного приложения и вызывают друг друга напрямую через методы/интерфейсы. Это является критичным, ведь нужно продумать переключение использования модуля на микросервис и обратно в случае проблем.
Основным модулем который реализует основную функцию(бронирование гостиницы) является BookingService. 

BookingService зависит от остальных модулей. Для осуществления бронирования он использует все остальные модули. Так же этот сервис осуществяет запись в БД.

Остальные сервисы запись в БД не осуществляют

PromoCodeService зависит от модуля UserService, ему необходима информация о пользователе для правильно определения валидности промокодов.

UserService, HotelService, ReviewService не зависимы от других модулей.

### <a name="_qmphm5d6rvi3"></a>**Решение**
Для достижения целевой архитектры применить паттерн **Strangler Fig** — выносить по одному компоненту, оставляя остальной функционал в монолите.

Каждый существующий модуль необходимо вынести в микросервис. Делить модули не имеет смысла, так каждый модуль несет ответсвенность только за одну бизнес функцию.

BookingService - модуль который вызывает все остальные модули монолита, зависит от них. Перенос этого модуля не повлияет на остальные модули и не потребует изменения остальных. В таком случае независимо от того, используется модуль BookingService монолита или микросервис монолит останется без изменений. 

Микросервис будет использовать REST api монолита для взаимодействия с модулем - внесение изменений в монолит не потребуется.

Аналогично  PromoCodeService который использует UserService.

Порядок реализации микросервисов
| | |
| :-: | :- |
| 1 | BookingService |
| 2 | PromoCodeService |
| 3 | UserService |
| 4 | HotelService |
| 5 | ReviewService |

Схема архитектуры платформы после переноса BookingService - [схема](../ADR/diagram/C4diaBookingService.puml)  

RESTgateway используется для маршрутизации запросов от frontend и Booking микросервиса. RESTgateway инкапсулирует взаимодействие/зависимости -  Booking микросервис не знает с кем именно он взаимодействует (монолит или микросервис), таким образом перенос модулей монолита в микросервисы не потребует изменений сушествующих микросервисов. 

ACL не используется, так как это большие трудозатраты, чем задокументировать форматы данных которые используют модули.

До переноса всех модулей в микросервисы использовать существующую БД. Когда перенос будет закончен, осуществить миграцию данных в хранилища микросервисов. 

План перехода: 
1. реализация шлюза
2. реализация микросервисов
3. миграция данных 
4. финальное тестирование и отключение монолита
5. внедрение kafka для реализации взаимодействия микросервисов
6. внедрение service mesh
7. внедрение метрик и аналитики



### <a name="_bjrr7veeh80c"></a>**Альтернативы**

В первую очередь вынести в отдельные микросервисы те модули которые имеют минимум зависимостей от других и не обрабатывают данные.
Таким образом UserService, HotelService, ReviewService подходят под условие - никак не обрабатывают данные и не обращаются к другим модулям.
В таком случае порядок переноса будет выглядеть так:
| | |
| :-: | :- |
| 1 | UserService |
| 2 | HotelService |
| 3 | ReviewService |
| 4 | PromoCodeService |
| 5 | BookingService |

**Недостатки, ограничения, риски**

BookingService и PromoCodeService используют остальные сервисы через вызов методов, поэтому после переноса используемых ими модулей в микросервисы, необходимо модифицировать соответственным образом монолит, причем таким образом что бы иметь возможность отката в случае проблем.
