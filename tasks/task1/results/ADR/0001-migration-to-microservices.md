# ADR-0001: Переход Hotelio от монолита к микросервисам

**Автор:** Артём  
**Дата:** 2026-04-27  
**Статус:** принято для первого этапа миграции

## Контекст

Hotelio сейчас реализован как Spring Boot монолит с одной PostgreSQL-базой. REST-контроллеры напрямую вызывают сервисы доменов, а сервисы работают с общей схемой БД через JPA-репозитории. Наиболее связанный сценарий - создание бронирования: `BookingService.createBooking()` проверяет пользователя, отель, отзывы, промокод, рассчитывает цену и сохраняет бронирование.

Ключевые проблемы текущей архитектуры:

| N | Проблема | Влияние |
|---|---|---|
| 1 | Все домены разворачиваются как один процесс | Нельзя независимо масштабировать бронирования в периоды пикового спроса |
| 2 | Общая база данных для users, hotels, bookings, promos, reviews | Сильная связанность схемы, риск блокировок и сложная миграция данных |
| 3 | `BookingService` синхронно зависит от четырех внутренних сервисов | Любой сбой зависимого модуля ломает основной revenue-сценарий |
| 4 | REST API монолита является единственной точкой интеграции | Сложно строить BFF и гибкие фронтенд-сценарии |
| 5 | Изменения требуют полного релиза монолита | Низкая скорость поставки и высокий риск регрессии |

## Функциональные требования

| N | Действующие лица или системы | Use Case | Описание |
|---|---|---|---|
| 1 | Клиент, партнерский канал | Создать бронирование | Клиент передает `userId`, `hotelId`, опциональный `promoCode`; система валидирует данные, рассчитывает скидку и сохраняет бронирование |
| 2 | Клиент, фронтенд | Получить список бронирований | Система возвращает все бронирования или бронирования конкретного пользователя |
| 3 | Booking service | Проверить пользователя | Перед созданием бронирования нужно проверить активность, blacklist и VIP-статус пользователя |
| 4 | Booking service | Проверить отель | Перед созданием бронирования нужно проверить доступность, заполненность и доверенность отеля |
| 5 | Booking service | Проверить промокод | Система проверяет срок действия и применимость промокода к пользователю |
| 6 | Аналитика | Получать историю бронирований | После создания бронирования событие должно быть доступно асинхронным потребителям |
| 7 | Фронтенд | Получить агрегированный личный кабинет | В целевом состоянии данные бронирований, отелей и промокодов должны агрегироваться через GraphQL/BFF |

## Нефункциональные требования

| N | Требование |
|---|---|
| 1 | Миграция должна быть поэтапной, без Big Bang-переписывания |
| 2 | REST-контракт монолита должен сохраняться для существующих клиентов на первом этапе |
| 3 | Новый сервис должен иметь собственную базу данных и владеть своими данными |
| 4 | Синхронные межсервисные вызовы должны иметь timeout, retry policy и circuit breaker |
| 5 | События домена должны публиковаться асинхронно через Kafka для аналитики и будущих read-models |
| 6 | Нужны health checks, структурные логи, метрики и distributed tracing |
| 7 | Переключение трафика должно поддерживать feature flag/canary и быстрый rollback |
| 8 | Нельзя дорабатывать старую доменную логику после миграции модуля; новая функциональность пишется в выделенном сервисе |
| 9 | Данные должны мигрироваться обратимо: backfill, сверка, затем переключение источника истины |

## Решение

Начать миграцию с выделения `BookingService` в отдельный `booking-service`.

Причины выбора:

| Критерий | Обоснование |
|---|---|
| Бизнес-критичность | Бронирование - основной revenue-сценарий Hotelio |
| Нагрузка | Пиковый трафик приходится на поиск и бронирование, а масштабировать только booking в монолите невозможно |
| Связанность | Создание бронирования зависит от users, hotels, promos и reviews, поэтому вынос первым позволяет сразу ввести явные контракты |
| Эволюция продукта | История бронирований, аналитика и личный кабинет естественно строятся вокруг событий бронирования |
| Соответствие следующим этапам | gRPC booking-service, Kafka history-service и GraphQL-кабинет являются логичным продолжением этой декомпозиции |

### Целевая архитектура ближайшего этапа

Диаграмма: `diagram/to-be-architecture.puml`.

В ближайшем целевом состоянии монолит остается публичной REST-точкой для совместимости, но создание бронирования перехватывается и маршрутизируется во внешний `booking-service` через gRPC. Новый сервис владеет отдельной `booking-db`, а данные users, hotels, promos и reviews читает через публичные REST API монолита как временный Anti-Corruption Layer. После успешного создания бронирования сервис публикует `BookingCreated` в Kafka, откуда `booking-history-service` строит аналитическую модель.

Дальнейшее целевое состояние:

| Домен | Целевое владение | Интеграции |
|---|---|---|
| Booking | Отдельный сервис и БД | gRPC/REST command API, Kafka events |
| User | Отдельный сервис и БД | Синхронная проверка статуса, события профиля |
| Hotel | Отдельный сервис и БД | REST/gRPC read API, пакетная загрузка для BFF |
| PromoCode | Отдельный сервис и БД | Проверка промокодов, override скидок в GraphQL |
| Review | Отдельный сервис и БД | Рейтинги и доверенность отеля, события отзывов |
| Frontend API | GraphQL Gateway/BFF | Federation поверх доменных субграфов |
| Platform | Service mesh, Kafka, observability | Canary rollout, retries, tracing, metrics |

### Strangler Fig-план

1. Зафиксировать текущее поведение монолита регрессионными тестами.
2. Добавить в монолит прокси/adapter для `POST /api/bookings`, управляемый feature flag: `local` или `external`.
3. Реализовать `booking-service` с контрактом gRPC и собственной PostgreSQL БД.
4. На первом этапе оставить `GET /api/bookings` в монолите, чтобы снизить риск миграции чтения.
5. Перенести запись бронирований в новый сервис: валидации users/hotels/promos/reviews выполнять через REST API монолита.
6. Выполнить backfill исторических бронирований из монолитной БД в `booking-db`, зафиксировать правила дедупликации по legacy id.
7. Включить shadow traffic или dual-run сверку для создания бронирования без влияния на клиентов.
8. Включить canary: 1%, 10%, 50%, 100% трафика на внешний сервис с метриками ошибок, latency и расхождений данных.
9. После стабилизации перевести `GET /api/bookings` на новый сервис или read model.
10. Запретить новые изменения booking-логики в монолите и удалить legacy-код после периода наблюдения.

### Контракты и данные

| Область | Решение |
|---|---|
| Команды бронирования | gRPC `CreateBooking` между монолитом и `booking-service`; REST-контракт для клиентов пока не меняется |
| Чтение зависимых данных | Временно REST-вызовы в монолит: users, hotels, promos, reviews |
| Владение данными | `booking-service` становится system of record для новых бронирований |
| Исторические данные | Backfill из таблицы `booking` монолита, затем сверка количества и контрольных сумм |
| События | Kafka topic `booking.created` с `bookingId`, `userId`, `hotelId`, `promoCode`, `discountPercent`, `price`, `createdAt` |
| Консистентность | Для аналитики допустима eventual consistency; для создания бронирования нужен синхронный ответ клиенту |

### Операционная модель

| Область | Решение |
|---|---|
| Наблюдаемость | Корреляционный `traceId` из REST-запроса монолита прокидывается в gRPC и Kafka headers |
| Надежность | Timeout, retry только для идемпотентных чтений, circuit breaker на зависимые сервисы |
| Rollback | Feature flag возвращает `POST /api/bookings` на локальный `BookingService` монолита |
| Релизы | Docker Compose локально, затем CI/CD с отдельным пайплайном booking-service |
| Безопасность | Внутренние вызовы ограничиваются сетевыми политиками; позже - mTLS через service mesh |

## Альтернативы

| Альтернатива | Плюсы | Минусы | Решение |
|---|---|---|---|
| Big Bang-переписывание всех модулей | Чистая архитектура сразу | Слишком высокий риск, долгая заморозка фич, сложно для маленькой команды | Отклонено |
| Сначала выделить UserService | Низкий риск, простой домен | Меньший эффект на масштабирование и бизнес-метрики, booking все еще остается узким местом | Отклонено как первый шаг |
| Сначала выделить HotelService | Полезно для поисковых сценариев | Создание бронирования продолжит зависеть от монолита и общей БД | Отклонено как первый шаг |
| Ввести API Gateway до выделения сервисов | Улучшает точку входа | Не решает связанность доменной логики и общей БД | Отложено |
| Выделить `booking-service` через gRPC и Kafka | Максимальный эффект на критичный сценарий, совместимо со Strangler Fig | Появляются сетевые вызовы, миграция данных и distributed failure modes | Принято |

## Недостатки, ограничения, риски

| Риск | Последствие | Митигация |
|---|---|---|
| Сетевые вызовы вместо прямых методов | Рост latency и новые точки отказа | Timeouts, circuit breakers, budgets latency, метрики p95/p99 |
| Разделение БД | Нет локальных транзакций между доменами | Четкое владение данными, события, idempotency, компенсации при необходимости |
| Расхождение данных при миграции | Ошибки в списках бронирований и аналитике | Backfill, сверка, read-only окно для legacy booking-таблицы на этапе переключения |
| Недостаток команды | Слишком много инфраструктурной работы | Первый этап ограничен booking-service; Kafka и observability вводятся минимально, но сразу по стандарту |
| Временная зависимость нового сервиса от монолита | Монолит остается критической зависимостью | Это осознанный промежуточный шаг; следующие домены выносятся по тем же контрактам |
| Неверный rollback после частичного создания | Дубли или потеря бронирования | Идемпотентный ключ запроса и дедупликация на стороне `booking-service` |

## Итог

Выбираем постепенную миграцию по Strangler Fig и первым выносим `BookingService`. На первом этапе внешний контракт для клиентов не меняется: монолит остается фасадом, но запись бронирований уходит в новый сервис с собственной БД. Это снижает риск миграции, дает независимое масштабирование главного сценария и создает основу для Kafka-аналитики, GraphQL/BFF и дальнейшего выделения доменных сервисов.
