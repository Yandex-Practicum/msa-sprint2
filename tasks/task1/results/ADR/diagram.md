### <a name="_qmphm5d6rvi3"></a>**Решение**
Приведите диаграммы контекста и контейнеров в модели C4. Опишите там основные компоненты и интеграции всех элементов решения.
Исходя из структуры монолита было принято решение внедрить деление по доменным областям и выделить следующие микросервисы:
- Сервис "Авторизация" - отвечает за аутентификацию и авторизацию пользователя
- Сервис "Пользователь" - отвечает за хранение данных пользователя, посредством репликации делит базу с сервисом авторизации.
- Сервис "Отели" - хранит данные об отелях.
- Сервис "Отзывы" - хранит данные об отзывах пользователей. 
  Опосредованно связан с сервисом пользователя и сервисом
  отеля(для добавления отзыва необходим user_id и hotel_id).
- Сервис "Промокоды" - хранит данные об промокодах.
- Сервис "Бронирование" - отвечает за бронирование отелей, получает данные от сервисов
- Брокер сообщений Kafka - отвечает за обмен сообщениями между сервисами.

Наиболее изолированные компоненты монолита на данный момент
От более изолированного к менее изовлированному.
1. Сервис - "Отели"
2. Сервис - "Промокоды"
3. Сервис - "Отзывы"
4. Сервис - "Пользователь" 
   (Сервис "Авторизация непосредственно связан с сервисом "Пользователь",
   поэтому в данном контексте не рассматривается).
5. Сервис - "Бронирование"
(при осуществлении бронирования осуществляются обращения к сервисам
"Пользователь", "Промокоды", "Отели", Отзывы).

С точки зрения изолированности начинать следует с сервиса "Отели",
однако, с точки зрения нагрузки наиболее нагруженым сервисом станет сервис авторизация,
так как любой запрос требует проверки прав пользователя.
Соответственно, при высоких нагрузках сервис авторизации потребует наибольшего числа
реплик.
Так же, несмотря на то, что от сервиса пользователь зависит значительное число сервисов ("Промокоды", "Отзывы", "Бронирование")
Сам сервис независим ни от одного из вышеописанных сервисов и может предоставить структурированное API для взаимодействия
с остальными сервисами.

Исходя из вышеперечисленного, первым к выделению в отдельный микросервис предлагается сущность "Пользователь"
с выделением отдельного микросервиса "Авторизация" с частичной репликацией базы данных пользователя с необходимыми, 
для авторизации данными.

Далее, двигаясь от менее более изолированной доменной области к менее изолированной, следует,
выделить сервисы "Отели", "Промокоды" и "Отзывы". Таким образом единственный сущностью, работающей
в монолите останется домен "Отели". Так как изменения в монолит будут внесены по ходу извлечения остальных сущностей,
разработчики будут переписывать запросы к другим доменным областям постепенно, избегая антипаттерна "Большой взрыв". 

Итогом станет превращение монолита в самостоятельный микросервис, взаимодействиующий с другими
сервисами через gRPC, что обусловленно необходимостью синхронного получения данных при осуществлении бронирования.
Однако, в сервисе "Бронирование" предполагается использование,
системы асинхронного выполнения задач и как следствие, увеличение скорости всей системы.

При нажатии кнопки забронировать, пользователь будет перенапрвлен на страницу с оповещением о начале
процедуры бронирования(дополнительно можно выводить этапы выполнения бронирования с сообщениям наподобие:
"Проверяем наличие мест", "Проверяем промокод", "Рассчитываем скидку", и т.д.).

В случае возникновения ошибки на одном из этапов, пользователь будет получать соответсвующее уведомление,
с указанием причины отказа, иначе сообщение об успешном завершении процесса
и все необходимые данные бронирования.

Такой подход позволит улучшить пользовательский опыт, и снизить время ожидания, за счёт возможности
репликации системы асинхронного выполнения задач, в случае увеличения нагрузки.
