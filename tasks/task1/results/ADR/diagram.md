### **Название задачи:**
Переход Hotelio с монолита на микросервисы (промежуточное состояние через 2-3 месяца)

### **Проблема**

Hotelio — сайт бронирования отелей, около 1 млн пользователей. Сейчас всё работает как монолит: один код, одна база данных, один сервер.

Что не так:

- **Сложно менять** — любое исправление требует понимания всей программы
- **Нельзя масштабировать** — при пике бронирований приходится масштабировать всё приложение целиком (и отзывы, и промокоды — хотя они не нужны)
- **Команды мешают друг другу** — нельзя работать независимо
- **API неудобный** — фронтенд получает те же данные, что и партнёры
- **Страшно выпускать новые фичи** — высокий риск сломать что-то рядом

Бизнес хочет постепенно переходить на микросервисы. Старые сервисы не трогаем, новое пишем только в вынесенных сервисах.

### **Функциональные требования**

| **№** | **Действующие лица** | **Use Case** | **Описание** |
|:---:|:---|:---|:---|
| 1 | Пользователь | Поиск отелей | Ищет отели по городу |
| 2 | Пользователь | Создание бронирования | Бронирует отель, может применить промокод |
| 3 | Пользователь | Просмотр бронирований | Смотрит свои брони |
| 4 | Пользователь | Отзыв | Оставляет отзыв об отеле |
| 5 | Администратор | Управление промокодами | Создаёт и отключает промокоды |

### **Нефункциональные требования**

| **№** | **Требование** |
|:---:|:---|
| 1 | Масштабировать можно отдельные сервисы (например, бронирования) |
| 2 | Если сломался сервис отзывов — бронирования всё равно работают |
| 3 | Сервисы не зависят друг от друга жёстко |
| 4 | Разные команды могут разрабатывать свои сервисы независимо |

### **Компоненты системы (через 2-3 месяца)**

| Компонент | Что делает |
|-----------|---------------------------------------------------------------------------------|
| **Booking Service** | Новый сервис. Всё про бронирования. Своя база данных. |
| **Монолит (остатки)** | Пользователи, отели, промокоды, отзывы. Старую логику не трогаем. |
| **BFF** | Прослойка между фронтендом и сервисами. Адаптирует API под клиентов. |
| **Kong API Gateway** | Единая точка входа. Маршрутизирует запросы: бронирования → в Booking Service, остальное → в монолит. Также занимается аутентификацией (JWT), лимитированием запросов, сбором логов и метрик. |
| **Kafka** | Для фоновых задач (начисление бонусов, обновление рейтинга) |

### **Решение**

#### Логика принятия решений

1. **Выносим по одному сервису** — Strangler Fig, требование от бизнеса

2. **Первым выносим Booking Service** — это самое загруженное и важное место. Именно здесь узкое место системы

3. **Booking Service общается с монолитом через gRPC** — так быстрее, строже контракт и проще генерировать клиента. Для синхронных вызовов (проверка пользователя, отеля, промокода) это подходит идеально. В будущем, когда эти сервисы тоже вынесем, gRPC уже будет готов

4. **Для фоновых задач используем Kafka** — например, начисление бонусов. Не критично по времени

5. **У каждого сервиса своя база** — начинаем с Booking Service (создаём `booking_db`). Это убирает главную проблему — общую базу данных

6. **BFF нужен** — чтобы фронтенд не зависел от того, как устроены сервисы внутри

7. **Kong API Gateway** — он даёт готовые плагины для аутентификации (JWT), лимитирования запросов, мониторинга. Это закрывает часть нефункциональных требований без дополнительной разработки

8. **Старый код не трогаем** — новые фичи пишем только в новых сервисах

### **Недостатки, ограничения, риски**

#### Недостатки

- Пока идёт миграция, нужно поддерживать и монолит, и новый сервис
- Запросы из нового сервиса в монолит работают чуть дольше (сеть)
- Данные могут временно расходиться между разными базами
- Kong становится единой точкой входа (нужно делать отказоустойчивым)

#### Ограничения

- **Нет распределённых транзакций** — если что-то пошло не так, данные могут быть не согласованы. Принимаем как данность
- **Монолит не трогаем вообще** — новые фичи делаем только в новых сервисах. Если нужна фича в пользователях, а User Service ещё не вынесли — ждём выноса или пересматриваем приоритеты. В монолит ничего не добавляем
- **Команда маленькая** (2 разработчика + архитектор + DevOps) — за 2-3 месяца успеем вынести 2-3 сервиса не больше

#### Основные риски

| Риск | Что делаем |
|------|------------|
| Kong сломался — вся система встала | Запускаем несколько реплик Kong, делаем резервирование |
| Booking Service долго отвечает (таймаут) | Ставим таймаут в Kong, настраиваем автоматические повторные попытки (retry) |
| Потеряли событие в Kafka | Настраиваем сохранение данных, проверяем, что сообщение обработалось |
| Команда не справляется с двумя системами | Автоматизируем сборку и деплой, строго проверяем готовность сервиса к выносу |
| gRPC сложнее настраивать, чем REST | Используем готовые инструменты (protoc, grpc-gateway при необходимости), документируем контракты |

### **Альтернативное решение**

Можно было начать не с Booking Service, а с User Service.

**Сравнение:**

| Что сравниваем | Booking Service (выбрали) | User Service (альтернатива) |
|:---|:---|:---|
| Ценность для бизнеса | Высокая — бронирования приносят деньги | Средняя — пользователи важны, но не напрямую |
| Помогает масштабировать | Да — самый нагруженный сценарий | Не очень — бронирования всё равно в монолите |
| Сложность реализации | Высокая — много зависимостей | Низкая — проще и независимее |
| Риск | Высокий | Низкий |

**Почему выбрали Booking Service:**

Бронирования — самое узкое место системы. Если не вынести их, проблема масштабирования не решится. Да, сложнее и рискованнее, но бизнес-эффект будет настоящий. К тому же, вынос сложного сервиса первым даст опыт и шаблон для всех остальных.

### **План миграции (Strangler Fig) — на 2-3 месяца**

#### Какой модуль выносим первым

**Booking Service**

**Почему:**
1. Самое загруженное место в системе
2. Много зависимостей — заставим правильно спроектировать API
3. Новые фичи по бронированиям сразу пойдут в новый сервис
4. Быстро покажем пользу от перехода

#### Этапы миграции

**1-2 недели. Проектирование и подготовка**
- Проектируем gRPC API для Booking Service (пишем .proto файлы)
- Создаём новую базу данных `booking_db`
- Инициализируем Booking Service (базово)
- Делаем gRPC адаптеры для вызовов в монолит (пользователи, отели, промокоды)

**3-4 недели. Запускаем понемногу**
- Поднимаем Kong API Gateway (с поддержкой gRPC)
- Настраиваем маршруты: `/api/bookings*` → Booking Service, остальное → монолит
- Направляем 5-10% запросов на новый сервис (проверяем, как работает)
- Смотрим на ошибки и скорость ответа

**5-6 недель. Переключаем полностью**
- Увеличиваем трафик до 50%, потом до 100%
- Старый код бронирований в монолите отключаем (только чтение)
- Всё новое пишем только в новом сервисе

**7-8 недель. Заканчиваем с Booking Service**
- Обновляем документацию
- Настраиваем сбор логов через Kong

**9-10 недель. Начинаем следующий**
- Выбираем User Service или Promo Service
- Повторяем всё то же самое

#### Что получим через 2-3 месяца

- Booking Service вынесен
- Второй сервис начали выносить
- Старые сервисы не трогаем
- Новые фичи только в новых сервисах
- Базы данных разделены (хотя бы частично)
- Kong управляет маршрутизацией, аутентификацией и лимитами
- Внутреннее общение между сервисами через gRPC (быстро и строго)
- Можно думать о переезде в облако
