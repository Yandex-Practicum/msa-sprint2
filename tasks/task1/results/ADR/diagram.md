### **Название задачи:**
Переход Hotelio с монолита на микросервисы (промежуточное состояние через 2-3 месяца)

### **Проблема**

Hotelio — сайт бронирования отелей, около 1 млн пользователей. Сейчас всё работает как монолит: один код, одна база данных, один сервер.

Что не так:

- **Сложно менять** — любое исправление требует понимания всей программы
- **Нельзя масштабировать** — при пике бронирований приходится масштабировать всё приложение целиком (и отзывы, и промокоды — хотя они не нужны)
- **Команды мешают друг другу** — нельзя работать независимо
- **API неудобный** — фронтенд получает те же данные, что и партнёры
- **Страшно выпускать новые фичи** — высокий риск сломать что-то рядом

Бизнес хочет постепенно переходить на микросервисы. Старые сервисы не трогаем, новое пишем только в вынесенных сервисах.

### **Нефункциональные требования**

| **№**   | **Требование**                                                  |
|:-------:|:----------------------------------------------------------------|
|    1    | Масштабировать можно отдельные сервисы (например, бронирования) |
|    2    | Если сломался сервис отзывов — бронирования всё равно работают  |
|    3    | Сервисы не зависят друг от друга жёстко                         |
|    4    | Разные команды могут разрабатывать свои сервисы независимо      |

### **Компоненты системы (через 2-3 месяца)**

| Компонент            | Что делает                                                           |
|----------------------|----------------------------------------------------------------------|
| **Booking Service**  | Новый сервис. Всё про бронирования. Своя база данных.                |
| **Монолит**          | Пользователи, отели, промокоды, отзывы. Старую логику не трогаем.    |
| **BFF**              | Прослойка между фронтендом и сервисами. Адаптирует API под клиентов. |
| **Apollo Gateway**   | Единая точка входа для GraphQL. Объединяет схемы субграфов.          |
| **Booking Subgraph** | GraphQL-адаптер над Booking Service для федерации.                   |
| **Hotel Subgraph**   | GraphQL-адаптер над монолитом (отели) для федерации.                 |
| **Kafka**            | Для фоновых задач (начисление бонусов, обновление рейтинга).         |

### **Решение**

#### Логика принятия решений

1. **Выносим по одному сервису** — Strangler Fig, требование от бизнеса

2. **Первым выносим Booking Service** — это самое загруженное и важное место. Именно здесь узкое место системы

3. **Booking Service общается с монолитом через gRPC** — так быстрее, строже контракт и проще генерировать клиента

4. **Для фоновых задач используем Kafka** — например, начисление бонусов. Не критично по времени

5. **У каждого сервиса своя база** — начинаем с Booking Service (создаём `booking_db`)

6. **GraphQL Federation через Apollo Gateway** — объединяет Booking Subgraph и Hotel Subgraph в единый API. Клиент получает данные из разных сервисов одним запросом

7. **CI/CD автоматизация** — GitLab CI пайплайн собирает Docker образы, тестирует и деплоит в Kubernetes

8. **Istio для управления трафиком** — канареечные релизы (90% на v1, 10% на v2), фича-флаги через заголовки, retry и circuit breaker

9. **Старый код не трогаем** — новые фичи пишем только в новых сервисах

### **Недостатки, ограничения, риски**

#### Недостатки

- Пока идёт миграция, нужно поддерживать и монолит, и новый сервис
- Запросы из нового сервиса в монолит работают чуть дольше (сеть)
- Данные могут временно расходиться между разными базами
- Apollo Gateway становится единой точкой входа (нужно делать отказоустойчивым)

#### Ограничения

- **Нет распределённых транзакций** — если что-то пошло не так, данные могут быть не согласованы
- **Монолит не трогаем вообще** — новые фичи делаем только в новых сервисах

#### Основные риски

| Риск                                     | Что делаем                                         |
|------------------------------------------|----------------------------------------------------|
| Gateway сломался — вся система встала    | Запускаем несколько реплик Gateway                 |
| Booking Service долго отвечает           | Ставим таймаут, настраиваем retry через Istio      |
| Потеряли событие в Kafka                 | Настраиваем сохранение данных, проверяем обработку |

### **Альтернативное решение**

Можно было начать не с Booking Service, а с User Service.

**Сравнение:**

| Что сравниваем          | Booking Service (выбрали)              | User Service (альтернатива)                  |
|:------------------------|:---------------------------------------|:---------------------------------------------|
| Ценность для бизнеса    | Высокая — бронирования приносят деньги | Средняя — пользователи важны, но не напрямую |
| Помогает масштабировать | Да — самый нагруженный сценарий        | Не очень — бронирования всё равно в монолите |
| Сложность реализации    | Высокая — много зависимостей           | Низкая — проще и независимее                 |
| Риск                    | Высокий                                | Низкий                                       |

**Почему выбрали Booking Service:**

Бронирования — самое узкое место системы. Если не вынести их, проблема масштабирования не решится. Да, сложнее и рискованнее, но бизнес-эффект будет настоящий.

### **План миграции (Strangler Fig) — на 2-3 месяца**

#### Этапы миграции

**1-2 недели. Проектирование и подготовка**
- Проектируем gRPC API для Booking Service (пишем .proto файлы)
- Создаём новую базу данных `booking_db`
- Инициализируем Booking Service (базово)
- Делаем gRPC адаптеры для вызовов в монолит (пользователи, отели, промокоды)

**3-4 недели. GraphQL Federation**
- Поднимаем Apollo Gateway
- Создаём Booking Subgraph (GraphQL над gRPC)
- Создаём Hotel Subgraph (GraphQL над монолитом)

**5-6 недель. CI/CD и деплой в Kubernetes**
- Настраиваем CI/CD
- Деплоим в Minikube через Helm

**7-8 недель. Istio и управление трафиком**
- Устанавливаем Istio в кластер
- Настраиваем

**9-10 недель. Заканчиваем с Booking Service**
- Обновляем документацию
- Настраиваем сбор логов через Kong

**9-10 недель. Начинаем следующий**
- Выбираем User Service или Promo Service
- Повторяем всё то же самое

#### Что получим через 2-3 месяца

- Booking Service вынесен
- Второй сервис начали выносить
- Старые сервисы не трогаем
- Новые фичи только в новых сервисах
- Базы данных разделены (хотя бы частично)
- Kong управляет маршрутизацией, аутентификацией и лимитами
- Внутреннее общение между сервисами через gRPC (быстро и строго)
- Можно думать о переезде в облако
