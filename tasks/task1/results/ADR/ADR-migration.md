### <a name="_hjk0fkfyohdk"></a>**Автор:** Колесников Сергей
### <a name="_uanumrh8zrui"></a>**Дата:** 31.08.2025

## 1. Контекст
Hotelio — онлайн-платформа для бронирования отелей и апартаментов. У неё около 1 млн зарегистрированных пользователей и основной трафик приходится на поиск и бронирование жилья. Система построена на Java-монолите, который обеспечивает API для клиентов и партнёрских каналов.<br/>
[Архитектура монолита](diagram/monolith-diagram.puml) <br/>
### Назначение модулей:
#### BookingService
Обрабатывает создание бронирований. Выполняет все проверки: пользователя, отеля, промокода, отзывов. Вычисляет финальную цену, скидку и сохраняет результат.
#### UserService
Проверяет статус пользователя: активен ли он, не находится ли в чёрном списке. И возвращает его статус.
#### HotelService
Предоставляет информацию об отелях. Используется для валидации бронирования и отображения информации в интерфейсе.
#### PromoCodeService
Проверяет промокоды: действительность, применимость к конкретному пользователю.
#### ReviewService
Отвечает за отзывы и рейтинг отеля. Влияет на возможность бронирования, если у отеля плохая репутация.

### Цели бизнеса
Начать поэтапный переход к микросервисной архитектуре, применяя паттерн Strangler Fig, — выносить по одному компоненту, оставляя остальной функционал в монолите. Это позволит упростить масштабирование, снизить время выхода фич, повысить отказоустойчивость и упростить независимую разработку разных команд.

### <a name="_qmphm5d6rvi3"></a>**Решение**
Изучив исходную архитектуру монолита можно выделить 5 доменных моделей. Далее на их основе провести миграцию от монолита к микросервисам:
- booking-service
- user-service
- hotel-service
- promo-service
- review-service

Первым будет вынесен сервис информации об отелях (hotel-service), т к это сервис в основном используется на чтение.
Вторым сервис отзывов, т к имеет связь только с сервисом бронирования.
Далее сервис пользователей и затем сервис промокодов.
В завершении доработать сервис бронирования под межсервисное взаимодействие.
Так же для удобства взаимодействия выделить gateway service. Это будет точка входа в систему. Данный сервис будет осуществлять аутентификацию/авторизацию пользователей, агрегировать данные из нескольких бэкенд-сервисов, преобразовывает их в нужный формат и отправляет на фронтенду, кешировать.

### <a name="_bjrr7veeh80c"></a>**Альтернативы**
Также возможен альтернативный вариант, где на порядок выделения микросервисов в монолит влияет бизнес необходимости заказчика.
Медленная/ненадежная работа личного кабинета и авторизации, что напрямую влияет на удовлетворенность пользователей сервисом.
В этом случае следует начать с вынесения функционала сервиса по работе с пользователями.
Следующим ограничением может быть необходимостью добавления новых фичей в сервис бронирования. 
Таким образом, переход будет осуществляться исходя из целей бизнеса.
