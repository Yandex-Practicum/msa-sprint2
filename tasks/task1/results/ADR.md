### <a name="_qmphm5d6rvi3"></a>**Решение**

Основной проблемой системы является отсутствие адаптации к изменчивым нагрузкам, при повышении пользовательской активности в туристический сезон
система (на текущем этапе) должна масштабироваться полностью, то есть невозможно повысить количество реплик отдельного компонента.
Так же из-за этого создается единая точка отказа приложения, при повышенной нагрузке на количество бронирований будут страдать и другие части приложения, 
такие как получение отзывов.

Соответственно Hotelio рекомендуется переехать на микросервисы, используя подход Strangler Fig, начиная выделять отдельные сервисы, а так же используя
ACL для сохранения целостности данных.

### <a name="_bjrr7veeh80c"></a>**Альтернативы**

В качестве альтернативы этому подходу можно предложить либо продолжать масштабироваться в рамках монолита, либо использовать другие подходы для миграции на микросервисы.

Рассмотрим их оба:
1) Монолит: можно поднимать определенное количество реплик за одним load-balancer'ом. Хоть это и решает вопросы масштабируемости в малых пропорциях, но все равно при
достаточно большом росте бизнеса этого может не хватить. Так же это требует куда больших вычислительных мощностей, чем масштабирование отдельных компонентов.
В этом подходе не разрешаются вопросы CI/CD и BFF. И сильная связанность компонентов будет мешать бизнесу расти.
2) Миграция по типу большого взрыва: подход даже по названию подразумевает огромные бизнес и технические риски. Если будет не хватать ресурсов и работа
к некоторому моменту будет завершена допустим на 60%, то на таком этапе приложение нельзя будет использовать. Плюс весь новый функционал написанный в процессе миграции
придется переписывать второй раз при переезде на микросервисы и повторно его тестировать.


### План миграции
1. Миграция начнется с выделения функционала, связанного с бронированиями в отдельный сервис (booking-service).
    Вся логика связанная с бронированиями должна хорошо масштабироваться, поскольку в туристический сезон активность будет явно больше 
    и система должна уметь стабильно работать в такие моменты.
2. Для обеспечения более надежной миграции должен быть создан слой ACL и API Gateway (/BFF), клиенты должны быть переведены на последний.
3. Далее по такому же процессу путем доработки ACL и выделением следующих сервисов должны быть выделены сервисы hotel-service, user-service, review-service.
    В зависимости от ожидаемого функционала booking-service может в себе содержать логику по работе с промо-кодами (поскольку она будет тесно связанна),
    аналогично hotel-service + review-service. Возможно для такой системы будет достаточно 3 микросервисов (user-service, hotel-service, booking-service)
4. После окончания перевода сервисов, постепенное удаление ACL
