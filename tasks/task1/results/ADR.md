# ADR-001: Целевая архитектура микросервисов Hotelio и выделение модуля «Отели»

### Название задачи
Целевая архитектура микросервисов Hotelio и выделение модуля «Отели» (Hotel Service) в независимый микросервис (Strangler Fig Phase 1)

### Автор
Системный аналитик / Tech Lead

### Дата
2026-05-12

### Статус
Предложено (Proposed)

---

## Текущая архитектура (AS-IS)

Hotelio представляет собой монолитное Java-приложение на Spring Boot, разбитое на внутренние модули: `hotel-service`, `booking-service`, `user-service`, `loyalty-service`, `notification-service`. Формально разделённые, они работают в едином процессе, используют общую реляционную базу данных (PostgreSQL) и взаимодействуют синхронно через прямые вызовы методов и JPA-репозитории.

Ключевые проблемы текущей архитектуры:
- **Сильная связанность данных:** Все модули читают/пишут в единую схему БД. Например, `LoyaltyService` напрямую обращается к `BookingRepository` для подсчёта числа бронирований, а `BookingService` вызывает `HotelRepository` для проверки доступности номеров. Это делает невозможным независимое изменение схемы данных и масштабирование отдельных модулей.
- **Синхронные цепочки вызовов:** Сценарий создания бронирования требует последовательных синхронных вызовов к модулям отелей, лояльности и уведомлений. Отказ любого из звеньев (например, сбой в расчёте скидок) блокирует весь сценарий — каскадный отказ.
- **Общий жизненный цикл:** Любое изменение в любом модуле требует сборки и развёртывания всего монолита. Релизные циклы разных команд жёстко связаны.
- **Сложность тестирования:** Интеграционные тесты поднимают весь контекст Spring, включая БД и очереди, что приводит к медленной обратной связи для разработчиков.

Для асинхронных уведомлений используется RabbitMQ (очередь `booking.notification`), однако гарантии доставки ограничены, а сам брокер не поддерживает долгосрочное хранение событий и их перечитывание.

## Целевая архитектура (TO-BE)

Целевая архитектура — набор независимых микросервисов, владеющих собственными данными и взаимодействующих асинхронно через события. Основные характеристики:

- **Декомпозиция по доменам:** Выделяются сервисы Hotel, Booking, Loyalty, User, Notification. Каждый сервис представляет собой отдельное Spring Boot приложение.
- **База данных на сервис (Database per Service):** Каждый сервис использует собственную схему или инстанс PostgreSQL. Это изолирует данные и позволяет командам независимо развивать схему и выбирать оптимальные стратегии индексирования/хранения. Перекрёстный доступ к чужим таблицам исключён — данные запрашиваются только через API или события.
- **Асинхронная событийная шина на Apache Kafka:** Все изменения состояния, важные для других сервисов, публикуются в виде доменных событий в Kafka. Сервисы подписываются на интересующие их топики и асинхронно обновляют своё состояние. Это разрывает синхронные цепочки вызовов и обеспечивает слабую связанность.
- **GraphQL для фронтового API (Backend for Frontend):** Внешние клиенты (веб, мобильные приложения) взаимодействуют с системой через единый GraphQL-эндпоинт. GraphQL решает проблему избыточной/недостаточной выборки данных, позволяя клиентам запрашивать ровно те поля, которые нужны для конкретного экрана. Это особенно важно в микросервисной архитектуре, где данные разнесены по разным сервисам — GraphQL-шлюз выполняет федерацию запросов, собирая данные из нескольких сервисов в один ответ без множественных круговых обходов (round-trips) клиента.
- **REST API для синхронного межсервисного взаимодействия:** Там, где асинхронная модель невозможна или нецелесообразна (например, проверка доступности номеров в реальном времени при создании бронирования), сервисы взаимодействуют синхронно через REST API с чёткими контрактами.
- **Кэширование нагруженных данных в Redis:** Для снижения нагрузки на базы данных и уменьшения задержек применяется Redis. Кэшируются редко изменяемые, но часто запрашиваемые данные — детали отелей, результаты поиска. Инвалидация кэша производится через события Kafka.

## Первый модуль для миграции: Hotel Service

### Почему Hotel Service

Для первого шага миграции по паттерну Strangler Fig выбран модуль отелей по следующим причинам:

1. **Автономность:** Hotel Service минимально связан с другими модулями. Его основная ответственность — CRUD отелей, управление номерами и удобствами, поиск. Он не участвует в критических бизнес-транзакциях вроде создания бронирования в качестве инициатора.
2. **Чёткие границы данных:** Доменные сущности (Hotel, Room, Amenity) естественно изолируются в отдельную схему БД. У других модулей нет легитимной потребности писать в эти таблицы напрямую.
3. **Высокая частота запросов:** Поиск и просмотр отелей — самые нагруженные операции в системе. Их выделение в отдельный сервис с собственным кэшем (Redis) немедленно снизит нагрузку на основной инстанс БД монолита и позволит независимо масштабировать сервис под пиковые нагрузки.
4. **Низкий риск:** В случае проблем с новым сервисом API Gateway может мгновенно переключить трафик обратно на монолит (функциональность Strangler Fig). Критический бизнес-поток создания бронирований не затрагивается на этом этапе.
5. **Быстрая победа:** Миграция Hotel Service демонстрирует работоспособность целевой архитектуры (Kafka, собственные БД, Redis) с минимальными рисками и создаёт шаблон для последующих миграций более сложных модулей.

### План миграции (Strangler Fig)

1. **Подготовка инфраструктуры:** Развернуть Kafka-кластер, настроить Debezium для захвата изменений таблиц отелей из основной БД в Kafka-топики.
2. **Создание Hotel Service:** Разработать независимый сервис с собственной схемой `hotel_db`. Реализовать REST API для CRUD и поиска отелей, gRPC для внутренней проверки доступности номеров. Подключить Redis для кэширования. Реализовать потребителя CDC-событий из Kafka для синхронизации `hotel_db` с изменениями, вносимыми через монолит.
3. **Теневое переключение:** Направить копию трафика `/api/v1/hotels/*` в новый сервис (traffic shadowing) для валидации ответов без влияния на пользователей.
4. **Канареечное развёртывание:** Переключать живой трафик постепенно (10% → 50% → 100%) с мониторингом ошибок и задержек.
5. **Полное переключение и очистка:** Перевести 100% трафика на Hotel Service. Отключить и удалить модуль отелей из монолита. Удалить соответствующие таблицы из основной БД, остановить CDC.

## Ключевые технологические решения

### Apache Kafka как событийная шина (вместо RabbitMQ)

Переход на Kafka обоснован фундаментальными различиями в модели хранения сообщений и возможностями платформы:

- **Перечитывание событий (Event Replay):** Kafka хранит сообщения в распределённом логе с настраиваемым временем хранения, не удаляя их после доставки. Это критически важно для возможности добавления новых сервисов в будущем (например, сервис аналитики), которым потребуется прочитать всю историю бронирований для инициализации состояния. RabbitMQ удаляет сообщения после подтверждения и не поддерживает такую возможность.
- **Строгий порядок обработки:** Партиционирование по ключу (например, `bookingId`) гарантирует, что события жизненного цикла бронирования (`created → confirmed → cancelled`) будут обработаны строго последовательно, без гонок.
- **Exactly-once семантика:** Идемпотентные продюсеры и транзакционная запись Kafka обеспечивают гарантии ровно однократной доставки, предотвращая дублирование критичных событий вроде начисления баллов лояльности или списания средств.
- **Экосистема:** Kafka Connect позволяет бесшовно интегрировать CDC (Debezium) без дополнительных компонентов. Kafka Streams обеспечивает потоковую обработку (например, онлайн-агрегация уровней лояльности) без развёртывания отдельных вычислительных кластеров.
- **Пропускная способность:** Kafka проектно рассчитана на миллионы сообщений в секунду, что исключает необходимость замены шины при масштабировании бизнеса.

### База данных на сервис (Database per Service)

Разделение баз данных — краеугольный камень микросервисной архитектуры. Выбор обоснован следующими причинами:

- **Изоляция схемы:** Каждая команда может независимо изменять структуру своих таблиц, добавлять индексы под свои сценарии, проводить миграции без координации с другими командами.
- **Изоляция нагрузки:** Ресурсоёмкие запросы поиска отелей не влияют на производительность критичных транзакций бронирования.
- **Полиглотное хранение:** В перспективе разные сервисы могут выбирать оптимальные технологии хранения (например, Elasticsearch для полнотекстового поиска отелей, PostgreSQL для транзакционных данных бронирований), а не быть привязанными к единой реляционной БД.
- **Отказоустойчивость:** Выход из строя базы данных одного сервиса не нарушает работу других.

### Redis для кэширования

- **Снижение нагрузки на БД:** Данные отелей и результаты поиска запрашиваются на порядки чаще, чем изменяются. Кэширование в Redis радикально снижает нагрузку на PostgreSQL.
- **Низкая латентность:** Redis обеспечивает предсказуемое субмиллисекундное время ответа, что критично для P99 задержки поискового API под нагрузкой.
- **Централизованная инвалидация:** При изменении отеля сервис публикует событие в Kafka, на которое подписан сам Hotel Service для сброса соответствующего кэша. Это гарантирует консистентность кэша между всеми экземплярами сервиса.

### GraphQL для фронтового API

- **Снижение числа запросов (Round-trips):** В микросервисной архитектуре данные для одной страницы (например, детали отеля + отзывы + похожие отели) могут находиться в разных сервисах. GraphQL-шлюз агрегирует их в один запрос, устраняя необходимость клиенту выполнять несколько HTTP-вызовов.
- **Контроль перевыборки/недовыборки:** Мобильные клиенты с ограниченным трафиком могут запрашивать только нужные поля (например, только название и цену для списка), в то время как десктоп запрашивает полное описание и галерею. REST с фиксированными DTO такого не обеспечивает.
- **Единая точка входа и типизация:** GraphQL-схема служит строго типизированным контрактом между фронтом и бэкендом, упрощая разработку клиентов и автогенерацию кода.

### REST API для синхронного межсервисного взаимодействия

Для случаев, где асинхронная модель неприменима (проверка доступности номера требует немедленного ответа "да/нет" в рамках сценария создания бронирования), используется REST:

- **Простота и прозрачность:** REST легко отлаживать (curl, Postman), стандартные коды ответов HTTP однозначно трактуются.
- **Зрелая экосистема:** Spring Web/MVC, OpenAPI-контракты, библиотеки клиентов (RestClient, Feign) глубоко интегрированы и проверены в продакшене.
- **Чёткое разделение:** GraphQL для фронтов (BFF), REST для межсервисных команд — каждая технология используется в своей нише, для которой она оптимальна.

## Альтернативы

### Оставить RabbitMQ как основную шину
Отклонено, так как RabbitMQ не поддерживает долгосрочное хранение событий (event replay) и не имеет встроенных примитивов потоковой обработки (Kafka Streams), что потребовало бы внедрения дополнительных компонентов для сценариев агрегации и аналитики.

### Мигрировать Loyalty Service первым
Отклонено из-за сильной циклической зависимости с Booking Service (скидки ↔ бронирования). Разрыв этой зависимости требует сложного распределённого оркестратора (Saga), что неоправданно рискованно для первого шага.

### Общая БД для всех микросервисов
Отклонено как архитектурный антипаттерн ("распределённый монолит"). Не решает проблему связанности данных и не позволяет командам развиваться независимо.

### Чистый REST API для фронтов
Отклонено в пользу GraphQL, так как REST приводит к проблеме избыточной/недостаточной выборки и требует множественных клиентских запросов для агрегации данных из разных сервисов.

## Риски

- **Сложность эксплуатации Kafka:** Требует специфических компетенций для настройки, мониторинга consumer lag, управления партициями. Mitigation: использовать managed-решения (если облако) или dedicated DevOps-специалиста.
- **Eventual Consistency:** В период миграции данные в монолите и новом сервисе могут расходиться на время задержки CDC. Mitigation: окно синхронизации минимизировано до <2 сек через Debezium, настроен алертинг.
- **Двойное бронирование (Race Condition):** Проверка доступности через REST/gRPC и последующая вставка бронирования неатомарны. Mitigation: оптимистическая блокировка (версия строки) с повторными попытками на стороне Booking Service.
- **Производительность GraphQL:** Сложные вложенные запросы могут создавать каскадные нагрузки на бэкенды. Mitigation: контроль глубины запросов, персистентные запросы (persisted queries), кэширование на уровне шлюза.
