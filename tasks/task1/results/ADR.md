### Ключевые проблемы
1. **Низкая возможность масштабируемости** - невозможно масштабировать отдельные компоненты
2. **Снижение надёжности по мере роста объёма кода** - ошибка в одном модуле влияет на всю систему
3. **Снижение скорости разработки по мере роста объёма кода** - длительные циклы тестирования и развертывания из-за необходимости тестирования всего приложения
4. **Технический долг** - накопленные проблемы, решение которых требует изменений в соседних модулях\компонентах

### Обнаруженные проблемы
- Отсутствие изоляции данных между модулями
- Нет четких границ контекстов (Bounded Context)
- Сложность тестирования отдельных компонентов

### **Решение**
За основу миграции на микросервисную архитектуру был взят паттерн Strangler Fig.
Первым шагом принято решение вынести UserService в отдельный микросервис со своей БД. 
Данный сервис был выбран из-за того, что у него низкая связность с другими компонентами монолита, не управляет сложной бизнес-логикой транзакций. Развертка данного микросервиса будет происходить рядом с монолитом через ACL. ACL в дальнейшем будет использоваться для коммуникации новых микросервисов с монолитом.

#### **План миграции**:

1. Настроить инфраструктуру
    - Развернуть систему оркестрации контейнеров (Kubernetes, Swarm, Openshift и т.п.)
    - Настроить CI/CD

2. Разработать микросервис:
    - Реализовать базовой функциональности
    - Мигрировать данные в отдельную БД
    - Настроить синхронизации данных старых пользователей в новую БД 

3. Внедрить API Gateway и ACL
    - Настроить маршрутизацию запросов между монолитом и микросервисом
    - Настроить маршрутизацию новых пользователей на новый сервис

4. Поэтапное переключение:
    - Только новые пользователи
    - Половина старых пользователей
    - Все пользователи в новый сервис

5. Отключение старого кода
    - Отключение синхронизации данных
    - Архивация старой БД

**Недостатки, ограничения, риски**

1. Повышается сложность E2E тестирования
2. Повышаются требования к квалификации разработчиков и администраторов
3. Усложняется поиск ошибок. Необходимо внедрение систем трассировки, логирования и мониторинга
4. Возможно увеличение времени ответа системы из-за ACL и других промежуточных обработчиков запроса
