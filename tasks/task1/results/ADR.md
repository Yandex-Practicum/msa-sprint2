### Функциональные требования

Верхнеуровневые Use Cases платформы Hotelio:

| **№** | **Действующие лица или системы** | **Use Case** | **Описание** |
| :-: | :- | :- | :- |
| 1 | Пользователь (клиент) | Создание бронирования | Пользователь отправляет `POST /api/bookings?userId=X&hotelId=Y&promoCode=Z`. Система проверяет: активность пользователя и отсутствие в блэклисте (AppUserService), операционность и доступность отеля (HotelService), рейтинг доверия к отелю — avg >= 4.0 и count >= 10 (ReviewService), валидность промокода с учётом VIP-статуса (PromoCodeService). Рассчитывается итоговая цена со скидкой. Результат сохраняется в БД. |
| 2 | Пользователь (клиент) | Просмотр истории бронирований | Пользователь запрашивает `GET /api/bookings?userId=X` и получает список своих бронирований: отель, промокод, скидка, итоговая цена, дата. |
| 3 | Пользователь (клиент) | Поиск отелей | Пользователь ищет отели по городу (`GET /api/hotels/by-city?city=Seoul`) или получает топ-рейтинговые отели (`GET /api/hotels/top-rated?city=Seoul&limit=5`). |
| 4 | Пользователь (клиент) | Просмотр информации об отеле | Пользователь получает детали отеля по ID: название, город, рейтинг, описание, статус (`GET /api/hotels/{id}`). |
| 5 | Пользователь (клиент) | Просмотр отзывов об отеле | Пользователь просматривает список отзывов о конкретном отеле (`GET /api/reviews/hotel/{hotelId}`) с рейтингом и текстом каждого отзыва. |
| 6 | Пользователь (клиент) | Проверка промокода | Пользователь проверяет действительность промокода (`GET /api/promos/{code}/valid?isVipUser=true`) или передаёт его при оформлении бронирования. |
| 7 | Пользователь (клиент) | Управление профилем / проверка статуса | Система проверяет, активен ли пользователь, в блэклисте ли он, имеет ли VIP-статус (`GET /api/users/{userId}`, `/status`, `/blacklisted`, `/vip`). |
| 8 | Аналитическая система | Сбор статистики бронирований | Внутренняя аналитика получает агрегированные данные по бронированиям (по пользователям, отелям, промокодам, временным периодам) — без нагрузки на боевую БД. |
| 9 | Фронтенд-приложение | Отображение личного кабинета | Фронтенд агрегирует данные из нескольких доменов (бронирования + отели + промокоды) для личного кабинета пользователя. Требует гибкого BFF-слоя с единой точкой запроса. |
| 10 | DevOps / Команды разработки | Независимый деплой сервисов | Разные команды должны иметь возможность независимо разрабатывать, тестировать и выкатывать компоненты системы без координации и даунтайма всей платформы. |

### Нефункциональные требования

| **№** | **Требование** |
| :-: | :- |
| 1 | **Масштабируемость**: BookingService является наиболее нагруженным модулем. При пиковых нагрузках (сезон отпусков, акции) система должна поддерживать горизонтальное масштабирование только этого компонента, не масштабируя весь монолит. |
| 2 | **Доступность**: Критический путь создания бронирования (`POST /api/bookings`) должен иметь SLA не ниже 99.9%. Отказ вспомогательного сервиса не должен приводить к полной недоступности платформы. |
| 3 | **Отказоустойчивость**: При недоступности вспомогательного сервиса (например, PromoCodeService) система должна деградировать gracefully. Circuit breaker и retry должны быть реализованы на уровне межсервисного взаимодействия. |
| 4 | **Согласованность данных**: В переходный период должна быть обеспечена согласованность данных. Eventual consistency приемлема для аналитических данных (Kafka). Транзакционные операции бронирования должны оставаться атомарными. |
| 5 | **Безопасность и авторизация**: ACL на уровне API — пользователь должен иметь доступ только к своим бронированиям. Межсервисное взаимодействие защищается mTLS через Istio в финальном состоянии. |
| 6 | **Наблюдаемость (Observability)**: Каждый микросервис должен предоставлять метрики (Prometheus), распределённую трассировку (Jaeger) и структурированные логи. Требование усиливается с каждой фазой миграции. |
| 7 | **Независимость деплоя**: Каждый выделенный микросервис разворачивается через собственный CI/CD-пайплайн без влияния на другие сервисы (rolling update, canary deployment). |
| 8 | **Ограниченность ресурсов команды**: Команда: 2 Java-разработчика, 1 архитектор (пишет код), 1 фронтенд-разработчик, 1 DevOps, 1 тестировщик, 1 PM. Не более 1–2 сервисов в миграции одновременно. |
| 9 | **Обратная совместимость API**: Внешний REST API монолита сохраняет обратную совместимость для клиентов в период миграции. Прокси-слой перенаправляет запросы к новым микросервисам прозрачно для клиентов. |
| 10 | **Изолированность данных**: В финальном состоянии каждый микросервис владеет своей БД. В переходный период — начать с разделения БД бронирований как наиболее нагруженной таблицы. |
| 11 | **Скорость разработки новых фич**: Новые функции после выделения сервиса разрабатываются исключительно в новом микросервисе. Монолит не дорабатывается для перенесённой функциональности. |
| 12 | **Технологический стек**: Java Spring Boot (микросервисы), PostgreSQL (БД), Apache Kafka (асинхронные события), Docker/Kubernetes (контейнеризация), gRPC (синхронная межсервисная коммуникация), GraphQL/Apollo Federation (BFF). |

---

### Решение

#### Анализ текущего состояния (As-Is)

Платформа Hotelio реализована как единое Java Spring Boot приложение. Все пять бизнес-модулей развёрнуты в одном процессе и используют одну общую базу данных PostgreSQL с таблицами: `app_user`, `booking`, `hotel`, `promo_code`, `review`.

**Граф зависимостей модулей монолита:**

```
BookingService ──► AppUserService   (проверка активности, блэклиста, VIP-статуса)
BookingService ──► HotelService     (проверка операционности, наличия мест)
BookingService ──► ReviewService    (проверка доверия: avg >= 4.0 AND count >= 10)
BookingService ──► PromoCodeService (валидация промокода, расчёт скидки)
PromoCodeService ──► AppUserService (проверка VIP-статуса для VIP-промокодов)
```

BookingService является центральным оркестратором — он вызывает все остальные сервисы. ReviewService содержит критическое бизнес-правило: отель можно бронировать только если у него не менее 10 отзывов со средним рейтингом >= 4.0.

**Основные архитектурные проблемы:**

1. Плотная связанность: изменение ReviewService требует полного перетестирования и переразвёртывания всего монолита.
2. Невозможность независимого масштабирования: BookingService — самый нагруженный модуль, но масштабируется только вместе со всем приложением.
3. Единая точка отказа: ошибка в любом компоненте приводит к недоступности всей платформы.
4. Блокировка параллельной разработки: команда не может работать над разными модулями независимо.
5. Shared database anti-pattern: все сервисы разделяют одну БД, создавая скрытые зависимости.
6. Прямой доступ аналитиков к боевой БД увеличивает нагрузку и создаёт риски безопасности.

Диаграмма текущего состояния: [`diagrams/01-context-as-is.puml`](diagrams/01-context-as-is.puml)

#### Выбор первого модуля для миграции: BookingService

Принципиальный вопрос при запуске Strangler Fig: с какого компонента начать. Рассмотрим все кандидаты по ключевым критериям:

| Критерий | BookingService | AppUserService | HotelService | PromoCodeService | ReviewService |
|---|---|---|---|---|---|
| Бизнес-критичность | **Критический** (revenue) | Высокая | Средняя | Средняя | Средняя |
| Пиковая нагрузка | **Наибольшая** | Средняя | Средняя | Низкая | Низкая |
| Потребность в независимом масштабировании | **Максимальная** | Нет | Нет | Нет | Нет |
| Входящих зависимостей (fan-in) | **0** | 2 | 1 | 1 | 1 |
| Исходящих зависимостей (fan-out) | 4 | 0 | 0 | 1 | 0 |
| Бизнес-эффект от выделения | **Максимальный** | Минимальный | Минимальный | Минимальный | Минимальный |

**Почему BookingService, несмотря на высокий fan-out?**

На первый взгляд более изолированные сервисы (AppUserService, ReviewService с fan-out = 0) кажутся проще для миграции. Однако их выделение даст минимальный бизнес-эффект — монолит продолжит страдать от тех же проблем масштабируемости и связанности.

Ключевые аргументы в пользу BookingService:

- **Fan-in = 0**: ни один другой модуль монолита не вызывает BookingService. Это означает, что прокси в монолите (`BookingController` → gRPC) не затрагивает других зависимостей.
- **Критический бизнес-путь**: именно создание бронирований генерирует выручку и испытывает пиковые нагрузки.
- **Готовый gRPC-контракт**: `booking.proto` уже присутствует в проекте, что ускоряет реализацию.
- **Управляемый fan-out**: на переходный период BookingService вызывает REST API монолита для данных других доменов — монолит становится агрегатором для остальных модулей.
- **Основа для дальнейшей миграции**: выделение BookingService создаёт прецедент и опыт для последующей миграции AppUserService, HotelService и других.

**Стратегия Strangler Fig применительно к BookingService:**

Монолит превращается в gRPC-прокси для операции создания бронирования. `BookingController.createBooking()` делегирует вызов на новый `booking-service` через gRPC (активируется через env-переменные `BOOKING_SERVICE_EXTERNAL_HOST` / `BOOKING_SERVICE_EXTERNAL_PORT`, которые уже присутствуют в `application.yml` и `docker-compose.yml` в закомментированном виде). Для клиентов API остаётся неизменным.

**Текущее состояние (As-Is) — C4 Context:**
[`diagrams/01-context-as-is.puml`](diagrams/01-context-as-is.puml)

Монолитное Spring Boot приложение, единая PostgreSQL БД. Аналитики читают напрямую из боевой БД — это проблема.

**Промежуточное состояние (~2 месяца) — C4 Container:**
[`diagrams/02-containers-intermediate.puml`](diagrams/02-containers-intermediate.puml)

- Монолит сохраняется, обслуживает домены: User, Hotel, PromoCode, Review.
- `booking-service` выделен как gRPC-микросервис с изолированной БД.
- `POST /api/bookings` в монолите → gRPC прокси → `booking-service`.
- `booking-service` → REST монолита для данных других доменов.
- `booking-service` → Kafka → `booking-history-service` → аналитическая БД.
- Аналитики читают из `history-db`, не из боевой БД.

**Финальное состояние (~1 год) — C4 Container:**
[`diagrams/03-containers-final.puml`](diagrams/03-containers-final.puml)

- 5 независимых микросервисов с изолированными БД.
- Apollo Federation (GraphQL Gateway) как BFF для фронтенда.
- Istio Service Mesh: mTLS, canary deployments, circuit breaker.
- Kafka как шина событий.
- Kubernetes с Helm-чартами для каждого сервиса.
- Observability: Prometheus + Grafana + Jaeger.

#### Пошаговый план миграции

**Фаза 1: Подготовка (Недели 1–2)**

1. Запустить приложение и базу данных (`docker compose up -d --build` в `tasks/task1/`).
2. Выполнить регрессионные тесты (`test/regress.sh`) — убедиться, что все эндпоинты работают.
3. Зафиксировать test-log.txt как baseline.
4. Добавить Kafka и Zookeeper в docker-compose (уже присутствуют в `hotelio-monolith/docker-compose.yml`).
5. Настроить Docker-сеть `hotelio-net` как основную для всех сервисов.

**Фаза 2: Выделение BookingService (Недели 3–6)**

1. Создать новый Spring Boot проект `booking-service`.
2. Реализовать gRPC-сервер на основе контракта `tasks/task2/booking.proto` (методы `CreateBooking` и `ListBookings`).
3. Перенести бизнес-логику из `BookingService.createBooking()` монолита в новый сервис. Данные других доменов получать через REST-вызовы к монолиту: `/api/users/{id}`, `/api/hotels/{id}`, `/api/promos/validate`, `/api/reviews/hotel/{id}/trusted`.
4. Создать изолированную PostgreSQL БД (`booking-db`) для `booking-service`.
5. Мигрировать данные: скопировать таблицу `booking` из монолитной БД в `booking-db` (shadow write + bulk copy).
6. Активировать gRPC-прокси в монолите через env-переменные.
7. Запустить регрессионные тесты — `POST /api/bookings` должен проходить через новый сервис.

**Фаза 3: Добавление booking-history-service (Недели 5–8)**

1. В `booking-service` добавить публикацию события `BookingCreated` в Kafka-топик `booking-events` после успешного создания бронирования.
2. Создать `booking-history-service`: Kafka consumer, подписанный на `booking-events`.
3. `booking-history-service` ведёт аналитическую БД (`history-db`) с агрегатами по пользователям, отелям, промокодам и временным периодам.
4. Перевести аналитиков на чтение из `history-db` вместо прямого доступа к боевой БД.

**Фаза 4: GraphQL Federation (Месяцы 2–4)**

1. Создать `booking-subgraph` (Node.js/Apollo): GraphQL схема бронирований с ACL — пользователь видит только свои бронирования (проверка `req.headers['userid']`).
2. Создать `hotel-subgraph`: GraphQL схема отелей с DataLoader для батчинга (решение N+1 проблемы при запросе `hotel { name }` для списка бронирований).
3. Создать `promocode-subgraph`: управляет промокодами, переопределяет поле `discountPercent` в схеме бронирований через `@override`.
4. Развернуть `Apollo Gateway` как единую точку входа для фронтенда, заменяя разрозненные REST-вызовы.

**Фаза 5: Kubernetes и CI/CD (Месяцы 3–5)**

1. Создать Helm-чарты для `booking-service`, `booking-history-service`, Apollo Gateway.
2. Реализовать CI/CD пайплайн (GitLab CI): стадии `build → test → deploy → tag`.
3. Настроить liveness и readiness пробы (`/ping`, `/health`) для каждого сервиса.
4. Реализовать два набора Helm-значений: `values-staging.yaml` и `values-prod.yaml`.
5. Настроить Service Discovery через DNS внутри кластера Minikube.

**Фаза 6: Service Mesh с Istio (Месяцы 4–6)**

1. Установить Istio в кластер Kubernetes (`istioctl install --set profile=demo`).
2. Настроить canary deployment для `booking-service`: 90% трафика на v1, 10% на v2.
3. Реализовать Circuit Breaker через `DestinationRule`.
4. Настроить fallback-маршрутизацию при ошибках (v1 → v2).
5. Добавить feature flags через `EnvoyFilter` (маршрутизация на v2 при заголовке `X-Feature-Enabled: true`).

**Фаза 7: Выделение оставшихся сервисов (Месяцы 6–12)**

1. Последовательно выделить `AppUserService`, `HotelService`, `ReviewService`, `PromoCodeService` в отдельные микросервисы с изолированными БД.
2. Перевести межсервисные вызовы от REST монолита к прямым gRPC/REST вызовам между микросервисами.
3. Настроить mTLS через Istio для всего межсервисного взаимодействия.
4. Добавить Prometheus + Grafana + Jaeger для всех сервисов.
5. Вывести монолит из эксплуатации.

#### Логика принятия решений и выбора технологий

**gRPC для интеграции монолит → booking-service:**
gRPC выбран как протокол интеграции, потому что контракт `booking.proto` уже определён в проекте. gRPC обеспечивает строгую типизацию через Protobuf, высокую производительность для межсервисных вызовов и нативную поддержку в Java Spring Boot через `grpc-spring-boot-starter`. Env-переменные для прокси (`BOOKING_SERVICE_EXTERNAL_HOST`, `BOOKING_SERVICE_EXTERNAL_PORT`) уже присутствуют в конфигурации монолита — нужно только раскомментировать.

**Kafka для booking-history-service:**
Аналитическая система не должна нагружать боевую транзакционную БД прямыми запросами. Kafka обеспечивает развязку производителя (booking-service) от потребителей без изменений в producer. Eventual consistency приемлема для аналитики. Kafka уже включена в `docker-compose.yml` монолита. Позволяет легко добавлять новых потребителей (сервис нотификаций, сервис рекомендаций) без изменений в `booking-service`.

**REST для вызовов booking-service → другие домены:**
На переходном этапе `booking-service` вызывает REST API монолита для получения данных пользователей, отелей, промокодов и отзывов. Это позволяет избежать дублирования бизнес-логики до момента полной миграции всех сервисов. Впоследствии эти вызовы заменятся прямыми gRPC-вызовами к независимым микросервисам.

**Apollo Federation / GraphQL:**
Фронтенд страдает от необходимости делать множество последовательных REST-запросов для агрегации данных из разных доменов. GraphQL Federation решает проблему N+1, предоставляя единую точку входа. DataLoader в `hotel-subgraph` батчит запросы к сервису отелей. `@override` в `promocode-subgraph` переопределяет поле `discountPercent` без изменения контракта `booking-subgraph`.

**Strangler Fig Pattern:**
Паттерн выбран как стратегия миграции, потому что: (1) монолит остаётся fallback на каждом шаге, (2) клиенты не замечают миграции — API сохраняется, (3) регрессионные тесты валидируют паритет на каждом этапе, (4) совместим с ограниченными ресурсами команды (7 человек).

### Альтернативы

#### Альтернатива 1: Big Bang Migration (полное переписывание)

**Описание:** Остановить монолит и за 3–4 месяца выпустить полностью переработанную микросервисную систему.

**Причины отклонения:**
- Критически высокий риск для бизнеса: платформа недоступна несколько месяцев.
- Команда из 7 человек не способна реализовать 5+ микросервисов с нуля за разумное время.
- Исторически высокий процент провала Big Bang миграций в production.
- Ошибки обнаруживаются только на финальном этапе — нет инкрементального тестирования.

#### Альтернатива 2: Начать с изолированного сервиса (AppUserService или ReviewService)

**Описание:** Выделить первым сервис без входящих зависимостей — как более простой с точки зрения интеграции.

**Причины отклонения:**
- Бизнес-эффект минимален: оба сервиса не являются узким местом по нагрузке.
- Монолит продолжает испытывать проблемы масштабируемости на критическом пути бронирования.
- Нарушает логику последующих задач проекта, которые предполагают BookingService как первый мигрированный компонент.
- Команда не получает опыта миграции сложного, центрального сервиса.

#### Альтернатива 3: Istio Service Mesh как первый шаг

**Описание:** Вместо gRPC-прокси — сразу развернуть Istio и использовать его маршрутизацию трафика.

**Причины отклонения:**
- Istio требует Kubernetes, что является дополнительной инфраструктурной сложностью на раннем этапе.
- DevOps на раннем этапе должен сосредоточиться на базовой контейнеризации.
- gRPC-прокси проще в реализации и достаточен для начального этапа.
- Istio планируется на финальном этапе (Фаза 6) — это соответствует итеративному подходу.

#### Альтернатива 4: GraphQL Federation с самого начала

**Описание:** Вместо gRPC-прокси использовать GraphQL Federation как первый слой интеграции.

**Причины отклонения:**
- Требует разработки нескольких subgraph-сервисов — значительный объём работы на старте.
- Node.js/Apollo Federation — другой технологический стек для Java-команды, требует обучения.
- gRPC обеспечивает лучшую производительность и строгую типизацию для server-to-server коммуникации.
- GraphQL оптимален как BFF для фронтенда, но избыточен для первоначальной межсервисной интеграции.

**Недостатки, ограничения, риски**

#### Недостатки выбранного решения

1. **Распределённые транзакции**: После выделения `booking-service` с собственной БД операция создания бронирования не может быть атомарной в рамках одной транзакции. При сбое после записи в `booking-db` (но до публикации в Kafka) возможна рассинхронизация. Необходима реализация паттерна Transactional Outbox или Saga.

2. **Временная рассинхронизация данных**: В переходный период таблица `booking` существует в двух местах (`monolith-db` и `booking-db`). `GET /api/bookings` в монолите читает из старой БД, а новые записи создаются в `booking-db`. Требует теневой записи или перевода чтения на новый сервис.

3. **Дополнительная сетевая задержка**: Каждый вызов `booking-service` → REST монолита добавляет сетевой round-trip. При нескольких последовательных вызовах (user → hotel → review → promo) суммарная задержка возрастает. Необходима параллелизация независимых проверок и circuit breaker с таймаутами.

4. **Временная зависимость booking-service от монолита**: На переходном этапе монолит остаётся точкой отказа для `booking-service`. Устраняется по мере выделения остальных сервисов.

5. **Сложность локальной разработки**: После появления нескольких сервисов локальная среда усложняется. Требуется стандартизация `docker-compose` конфигурации для всей команды.

#### Ограничения

1. **Ресурсы команды**: 2 Java-разработчика + архитектор не могут параллельно вести разработку более чем 1–2 сервисов. Скорость миграции: 1 сервис за 4–6 недель.
2. **Кривая обучения**: Потребуется время на освоение gRPC, Kafka и Kubernetes. Рекомендуется 1–2 недели на POC перед началом разработки каждого нового инструмента.
3. **Монолит как единая точка отказа**: Несмотря на выделение BookingService, монолит обслуживает остальные домены. Его недоступность влияет на booking-service через REST-вызовы.
4. **Сложность миграции данных**: Перенос таблицы `booking` требует тщательно спланированной стратегии с планом отката.
