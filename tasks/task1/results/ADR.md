### <a name="_u8xz25hbrgql"></a>**Нефункциональные требования**
Опишите здесь нефункциональные требования и архитектурно значимые требования.

|**№**|**Требование**|
| :-: | :- |
|1|**Производительность**: Время отклика поиска < 300ms, бронирования < 3s|
|2|**Масштабируемость**: Возможность независимого масштабирования компонентов под нагрузкой|
|3|**Доступность**: 99.9% для критичных сервисов (поиск, бронирование, платежи)|
|6|**Поддержка**: Возможность независимой разработки и деплоя команд|

### <a name="_qmphm5d6rvi3"></a>**Решение**

1. **API Gateway** - единая точка входа для управления трафиком, безопасности и маршрутизации
2. **Database per Service** - изоляция данных, независимое масштабирование, выбор оптимальной СУБД
3. **Event-Driven Architecture** - слабая связанность через события (Kafka)

### <a name="_bjrr7veeh80c"></a>**Альтернативы**
Горизонтальное масштабирование, запуск нескольких версий монолитного приложения.

**Недостатки, ограничения, риски**

1. **Сложность распределенных систем**
   - Достижение консистентного состояния данных
   - Разработка компенсирующих мероприятий в случает отказов отдельных компонентов

2. **Операционные затраты**
   - Увеличение инфраструктурных затрат
   - Сложность мониторинга и управления

3. **Миграция данных**
   - Риск потери данных при разделении БД

4. **Сетевые задержки**
   - Увеличение latency за счет межсервисной коммуникации
   - Зависимость от надежности сети

5. **Командные изменения**
   - Необходимость переобучения команд
   - Изменение процессов разработки и DevOps
