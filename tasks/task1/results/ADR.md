### <a name="_3bfxc9a45514"></a>**Функциональные требования**

|**№**|**Действующие лица или системы**|**Use Case**|**Описание**|
| :-: | :- | :- | :- |
| 1 | Пользователь | Создание бронирования | Пользователь отправляет запрос на бронирование отеля с параметрами: даты, тип номера, промокод. Система валидирует данные и создаёт запись о бронировании. |
| 2 | Booking Service | Валидация пользователя | Сервис запрашивает у UserService статус пользователя (активен/в чёрном списке) перед созданием бронирования. |
| 3 | Booking Service | Валидация отеля | Сервис запрашивает у HotelService информацию о доступности номеров и актуальной цене. |
| 4 | Booking Service | Применение промокода | Сервис отправляет запрос в PromoCodeService для проверки валидности и расчёта скидки. |
| 5 | Booking Service | Проверка репутации отеля | Сервис запрашивает у ReviewService рейтинг отеля для блокировки бронирования при низком рейтинге. |
| 6 | Booking Service | Расчёт финальной стоимости | Сервис вычисляет итоговую цену с учётом скидок, налогов и динамического ценообразования. |
| 7 | Booking Service | Сохранение бронирования | Сервис сохраняет бронирование в своей БД с статусом `PENDING` или `CONFIRMED`. |
| 8 | Booking Service | Публикация события | После успешного создания бронирования сервис публикует событие `BookingCreated` в Kafka для асинхронной обработки. |
| 9 | Партнёрские системы | Получение уведомлений | Внешние системы подписываются на события бронирования через Kafka для синхронизации данных. |

### <a name="_u8xz25hbrgql"></a>**Нефункциональные требования**
Опишите здесь нефункциональные требования и архитектурно значимые требования.

|**№**|**Требование**|
| :-: | :- |
| 1 | **Масштабируемость**: Возможность горизонтального масштабирования Booking Service независимо от монолита при росте нагрузки. |
| 2 | **Доступность**: Уровень доступности сервиса ≥ 99.9% (SLA), так как бронирование — критическая бизнес-функция. |
| 3 | **Производительность**: Время ответа эндпоинта `POST /api/bookings` ≤ 500 мс (p95) при нормальной нагрузке. |
| 4 | **Согласованность данных**: Обеспечение транзакционной целостности внутри Booking Service (ACID), eventual consistency при взаимодействии с монолитом. |
| 5 | **Отказоустойчивость**: При недоступности сервисов монолита Booking Service должен корректно обрабатывать ошибки (Circuit Breaker, Retry). |
| 6 | **Безопасность**: Аутентификация и авторизация запросов через JWT, валидация входных данных, защита от инъекций. |
| 7 | **Наблюдаемость**: Логирование, метрики (Prometheus) и распределённая трассировка (OpenTelemetry) для всех внешних вызовов. |
| 8 | **Совместимость**: Обратная совместимость API для существующих клиентов (frontend, партнёры) в период миграции. |

### <a name="_qmphm5d6rvi3"></a>**Решение**

**Диаграмма архитектуры:**
[C3.puml](C3.puml)

**Описание решения:**
1.  **Выделение сервиса:** Booking Service выделяется в отдельный контейнер Docker с собственным циклом жизни.
2.  **База данных:** Реализуется паттерн *Database per Service*. Booking Service получает отдельную схему/БД PostgreSQL. Данные пользователей, отелей и промокодов остаются в монолите на переходный период.
3.  **Интеграция:**
    *   **Синхронная:** Для валидаций (User, Hotel, Promo, Review).
    *   **Асинхронная:** Для уведомления других систем о успешном бронировании используется Kafka (топик `booking.created`).
4.  **Технологии:** Spring Boot 3, Spring Cloud (для клиентов и resilience), PostgreSQL, Kafka, Docker.

**Логика принятия решений:**
*   **Почему Booking Service первым?** Это наиболее нагруженный и бизнес-критичный модуль. Его изоляция даст максимальный выигрыш в производительности и отказоустойчивости.
*   **Почему своя БД?** Чтобы исключить блокировки таблиц монолита при пиковых нагрузках на бронирования и обеспечить независимое масштабирование.

### <a name="_bjrr7veeh80c"></a>**Альтернативы**

1.  **Modular Monolith (Spring Modulith)**
    *   *Суть:* Выделить модуль Booking внутри монолита с чёткими границами, но без физического разделения.
    *   *Почему не выбрано:* Не решает проблему операционного масштабирования и единой точки отказа (БД монолита).

2.  **Shared Database (Общая БД)**
    *   *Суть:* Новый сервис подключается к той же БД, что и монолит.
    *   *Почему не выбрано:* Высокий риск блокировок, нарушение инкапсуляции данных, сложность будущего разделения. Допустимо только как временная мера на 1-2 спринта.

3.  **Полный Event-Driven (Saga)**
    *   *Суть:* Сразу реализовать оркестрацию через события для всех валидаций.
    *   *Почему не выбрано:* Избыточная сложность для первого шага миграции. Требует глубокого рефакторинга монолита. Отложено на второй этап.

**Недостатки, ограничения, риски**

| Тип | Риск/Недостаток | Влияние | Митигация |
| :-: | :- | :- | :- |
| **Технический** | Сетевые задержки при вызовах к монолиту | Увеличение времени ответа | Внедрение Circuit Breaker, кэширование справочных данных |
| **Технический** | Рассинхронизация данных между БД | Ошибки валидации | Eventual consistency, периодические джобы сверки |
| **Операционный** | Усложнение деплоя и мониторинга | Рост затрат на DevOps | Использование единых CI/CD пайплайнов, централизованный логинг |
| **Бизнес** | Регрессии в логике бронирования | Потеря заказов | Полное покрытие тестами, Canary-релизы, быстрый откат |
| **Архитектурный** | Временная зависимость от монолита | Монолит остаётся узким местом | План постепенного выноса зависимых сервисов (User, Hotel) |
