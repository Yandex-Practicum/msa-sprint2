Стратегия миграции Hotelio на микросервисы

Дата создания: 2025-08-29

Обзор проекта

Hotelio - это онлайн-платформа для бронирования отелей, которая в настоящее время работает как монолитное приложение на Java Spring Boot. Проект направлен на постепенную миграцию к микросервисной архитектуре, используя паттерн Strangler Fig.

Текущая архитектура

Монолит
Технология: Java 17 + Spring Boot + Gradle
База данных: PostgreSQL
Функциональность: 
  Управление пользователями
  Управление отелями
  Управление промокодами
  Управление отзывами
  Управление бронированиями (мигрировано)

Kong Gateway
Роль: Внешний прокси-маршрутизатор
Преимущество: НЕ изменяет код монолита
Функциональность: Маршрутизация запросов между сервисами

Новая архитектура

Микросервисы

1. Booking Service
Порт: 8085 (внутренний для Kong)
База данных: booking_service (PostgreSQL, порт 5433)
Функциональность: Создание и управление бронированиями
Особенность: Отправляет события в Kafka

2. Booking History Service
Порт: 8086 (внутренний для Kong)
База данных: booking_history (PostgreSQL, порт 5434)
Функциональность: Аналитика и статистика бронирований
Особенность: Читает события из Kafka

Поток данных

Frontend -> Kong Gateway -> [Маршрутизация] -> Соответствующий сервис
    ↓           ↓              ↓                    ↓
  REST      Прокси      Правила маршрутизации    Бизнес-логика
  Запрос    Kong        • GET /bookings -> монолит
                        • POST /bookings -> booking-service
                        • /api/statistics -> history-service

Ключевое преимущество: Kong Gateway

Что мы НЕ делаем
 НЕ изменяем код монолита - вообще никак
 НЕ добавляем прокси-классы в Spring Boot
 НЕ модифицируем существующие сервисы
 НЕ нарушаем принцип единственной ответственности

Что мы делаем
 Добавляем Kong Gateway как внешний прокси
 Настраиваем маршрутизацию через конфигурацию
 Сохраняем чистоту кода монолита
 Обеспечиваем гибкость маршрутизации

Техническая реализация

Структура Kong конфигурации

# kong.yml
services:
  # Монолит - все GET запросы и остальные API
  - name: monolith-service
    url: http://monolith:8080
    routes:
      - name: monolith-bookings-get
        paths: ["/api/bookings"]
        methods: ["GET"]
      
      - name: monolith-other-apis
        paths: ["/api/users", "/api/hotels", "/api/promocodes", "/api/reviews"]

  # Booking Service - только POST /api/bookings
  - name: booking-service
    url: http://booking-service:8080
    routes:
      - name: booking-create-route
        paths: ["/api/bookings"]
        methods: ["POST"]

  # History Service - вся статистика
  - name: history-service
    url: http://booking-history-service:8081
    routes:
      - name: history-statistics
        paths: ["/api/statistics"]

Порты и доступность

| Сервис | Внешний порт | Внутренний порт | Описание |
|--------|--------------|-----------------|----------|
| Kong Gateway | 8000 | - | Основной API Gateway |
| Kong Admin | 8001 | - | Администрирование Kong |
| Монолит | 8084 | 8080 | Прямой доступ (для отладки) |
| Booking Service | 8085 | 8080 | Прямой доступ (для отладки) |
| History Service | 8086 | 8081 | Прямой доступ (для отладки) |

Стратегия миграции

Этап 1: Kong Gateway (текущий)
 Добавление Kong без изменения монолита
 Настройка маршрутизации для бронирований
 Тестирование новой архитектуры
 Валидация корректности данных

Этап 2: Расширение микросервисов
 Миграция UserService через Kong
 Миграция HotelService через Kong
 Миграция PromoCodeService через Kong
 Миграция ReviewService через Kong

Этап 3: Оптимизация
 Переход на gRPC между сервисами
 Внедрение Service Mesh (Istio)
 Мониторинг и метрики через Kong
 Безопасность и аутентификация

Преимущества подхода с Kong

Архитектурные преимущества
1. Чистота кода - монолит остается нетронутым
2. Разделение ответственности - маршрутизация отдельно от бизнес-логики
3. Гибкость - легко менять маршруты без пересборки
4. Масштабируемость - Kong может масштабироваться независимо

Операционные преимущества
1. Zero-downtime - изменение маршрутов на лету
2. A/B тестирование - направление части трафика на новые сервисы
3. Canary deployments - постепенное развертывание
4. Откат - мгновенный возврат к старой конфигурации

Технические преимущества
1. Профессиональный подход - Kong используется в продакшне
2. Встроенные плагины - rate limiting, CORS, аутентификация
3. Мониторинг - детальная аналитика запросов
4. Безопасность - защита от атак на уровне Gateway
