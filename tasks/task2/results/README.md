# Стратегия миграции и целевая архитектура (Booking Service)

## 🔄 Стратегия миграции данных

1. **Dual-Write (параллельная запись)**  
   Новый `booking-service` пишет данные в свою БД и публикует событие `BookingCreated` в Kafka. Монолит временно сохраняет данные для обратной совместимости.

2. **Backfill (исторические данные)**  
   Существующие бронирования из монолита выгружаются и публикуются в Kafka с флагом `is_historical: true` для заполнения аналитики.

3. **Cutover (переключение трафика)**  
   Постепенный перевод запросов на gRPC-интерфейс нового сервиса с мониторингом latency, error rate и Kafka lag.

4. **Decommission (завершение)**  
   После стабилизации — удаление дублирующей логики из монолита и архивация старых таблиц.

## 🎯 Целевая архитектура (To Be)

Client → API Gateway → booking-service (gRPC) → PostgreSQL
↓
Kafka → booking-history-service → Analytics DB

| Компонент | Технология | Ответственность |
|-----------|-----------|----------------|
| `booking-service` | gRPC + PostgreSQL | CRUD бронирований, валидация, публикация событий |
| `booking-history-service` | Kafka Consumer | Асинхронная аналитика и отчёты |
| `monolith` | REST (legacy) | Fallback и постепенная передача функциональности |

## 🔑 Ключевые решения

- **Асинхронность**: Статистика считается отдельно через Kafka, без нагрузки на боевую БД.
- **Идемпотентность**: Потребитель обрабатывает события по `booking_id`, чтобы избежать дублей.
- **Транзакционный outbox**: Событие публикуется в Kafka в той же транзакции, что и запись в БД.
- **Откат**: При проблемах трафик автоматически возвращается в монолит через circuit breaker.
