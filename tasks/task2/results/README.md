# Task 2 — Миграция booking-логики из монолита в микросервис

## Что сделано

В рамках задания логика создания бронирования вынесена из монолита в отдельный микросервис `booking-service`.  
Монолит больше не создает бронирование напрямую в своей БД, а перенаправляет запрос `create booking` в новый gRPC-сервис.  
При этом остальные данные, необходимые для проверки бизнес-логики, по-прежнему берутся из монолита через REST-вызовы.  
Отдельно реализован сервис `booking-history-service`, который асинхронно получает события о создании бронирований через Kafka и сохраняет исторические данные в свою БД.

## Стратегия миграции данных при запуске нового сервиса

Для новой booking-логики используется отдельная БД микросервиса `booking-service`.  
Исторические данные из монолита автоматически не переносятся, так как в рамках задания основная цель — отделить новую логику создания бронирований и перевести новые записи в новый контур.  
После запуска новой архитектуры все новые бронирования создаются уже в БД `booking-service`.  
Дополнительно каждое успешное создание бронирования публикуется как событие `BookingCreated` в Kafka, после чего `booking-history-service` асинхронно сохраняет запись в таблицу истории.  
Такой подход позволяет не нагружать боевую БД аналитическими и историческими запросами.

## To Be стратегия

Целевая схема работы после миграции выглядит так:

- монолит принимает HTTP-запрос от клиента;
- запрос на создание бронирования перенаправляется в `booking-service` по gRPC;
- `booking-service` выполняет всю booking-бизнес-логику;
- необходимые данные о пользователе, отеле, отзывах и промокодах сервис получает из монолита по REST;
- новое бронирование сохраняется в собственной БД `booking-service`;
- после сохранения публикуется событие `BookingCreated` в Kafka;
- `booking-history-service` читает событие и сохраняет исторические данные в отдельную БД;
- запросы на чтение списка бронирований пока остаются совместимыми с текущим API монолита.

## Почему выбрана такая стратегия

Выбран постепенный сценарий миграции без полного переписывания системы.  
Он позволяет вынести только один бизнес-домен booking не затрагивая остальные части монолита.  
gRPC используется как новый внутренний контракт между монолитом и booking-сервисом, а Kafka - для асинхронной фиксации истории бронирований.  
Такой подход снижает риски миграции, позволяет поэтапно переносить функциональность и подготавливает систему к дальнейшему разбиению на микросервисы.
