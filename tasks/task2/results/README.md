# Task2 - Стратегия миграции данных и целевое состояние (To‑Be)

Дата: 2026‑03‑25

## 1. Объём этапа (Scope)
- Выделен сервис `booking-service` (gRPC) с собственной БД `booking` (PostgreSQL).
- Добавлен `booking-history-service` - консьюмер Kafka (topic `booking.created`) с БД `history`.
- Монолит остаётся входной точкой REST. Монолит проксирует создание бронирований в gRPC (`BOOKING_SERVICE_EXTERNAL_HOST/PORT`).
- Чтение списка бронирований в монолите остаётся как раньше (читает свою БД) - перенос чтения запланирован на следующий этап.

## 2. Текущее поведение (As‑Is после Task2)
- `POST /api/bookings` (монолит):
  1) Монолит валидирует входные параметры и вызывает gRPC `CreateBooking` в `booking-service`.
  2) `booking-service` повторно валидирует (обращаясь к монолиту по REST), считает скидку/цену, пишет запись в свою БД `booking` и публикует событие `BookingCreated` в Kafka.
  3) Монолит сохраняет отражение бронирования в свою таблицу и возвращает REST‑ответ (согласованный с gRPC).
- `GET /api/bookings?userId=...` (монолит): читает только БД монолита (специально не меняли).
- `booking-history-service`: читает Kafka и асинхронно пишет ту же запись в `history.booking_history` (идемпотентно `ON CONFLICT DO NOTHING`).

## 3. Целевое состояние этого этапа (To‑Be v1)
- Источник записей бронирований - `booking-service` (гарантирует запись в свою БД и публикацию события).
- Монолит остаётся совместимым и продолжает поддерживать REST API для клиентов.
- История формируется из Kafka без обращений к боевой БД.

Схема потоков данных:
- Client → (REST) Monolith → (gRPC) BookingService → (DB booking) + (Kafka booking.created) → BookingHistoryService → (DB history)
- Для чтения: Client → (REST) Monolith → (DB monolith)

## 4. Стратегия миграции данных
- Стартовая миграция: **dual‑write** - запись бронирования отражается и в БД монолита, и в БД `booking-service` (естественный результат текущей интеграции). История наполняется из Kafka.
- Обратной синхронизации не требуется на этом шаге: монолитные чтения используют собственную таблицу, аналитика - `history`.
- Контроль целостности: сравниваем id/кол-во строк в `booking` и `booking_history` после тестов (см. `regress.sh`).

План наполнения/обратного заполнения (если понадобится в следующем шаге):
- Одноразовый backfill старых бронирований из монолита → `booking-service` и/или в `history` (через batch + публикацию событий или прямые INSERT с сохранением id).

## 5. Наблюдаемость и ошибки
- Матчинг логов: `booking-service` пишет «gRPC booking-service listening…», `booking-history-service` - «Saved booking history <id>».

## 6. Что будет на следующем этапе (To‑Be v2, план)
1) Перевести чтение списка бронирований в монолите на gRPC `ListBookings` (с пагинацией/фильтрацией). 
2) Декомиссия монолитной таблицы `booking` (после cut‑over чтения и backfill). 
3) Добавить соглашения о схеме истории: `event_id`, `received_at`, partition/offset для аудита. 
4) Нормализовать коды ошибок и договор о контракте (grpc <-> REST). 
5) Включить ретраи/алерты на публикацию/консьюм в Kafka.

## 7. Как воспроизвести и проверить
- Запуск: `docker compose -f tasks/task2/docker-compose.yml up -d --build`
- Регресс‑скрипт: `bash tasks/task2/results/regress.sh` (Git Bash/WSL). 
  - Логи команд: `tasks/task2/results/test-log.txt`
  - Снимки таблиц: `select-booking-db.txt`, `select-history-db.txt`, `select-monolith-db.txt`
- Визуальные проверки:
  - REST‑листинги монолита показывают новые id.
  - gRPC листинги (grpcurl или встроенный клиент) показывают те же записи.
  - В `history` появились такие же id (исторические записи).

## 8. Риски и ограничения этого шага
- Временная **дубликация данных** (монолитная БД и БД сервиса) - принимается до cut‑over чтений.
- Возможна **рассинхронизация** при частичных сбоях (успех REST, но падение Kafka/consumer). Смягчается идемпотентной записью в историю и ретраями.
- Путь к proto/томам под Windows (Git Bash) требует `MSYS_NO_PATHCONV=1` - учтено в `regress.sh`.
