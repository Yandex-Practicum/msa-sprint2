
Стратегия миграции данных при запуске нового сервиса:
- Выделить сущность "booking" в отдельный сервис.
- Разработать сервис "booking_history".

Стратегия To be:
- Сущньсть "Бронирование" и вся связанная с ней логика была вынесена в отдельный сервис "booking".
- Взаимодействие между сервисом "booking" и монолитом осуществляется посредством gRPC.
- В текущей версии запросы к монолиту/сервисам пользователь/отель/промокод/отзывы не реализованы,
поэтому проверки пользователей, отелей и рассчёт скидки осуществляются условно. Например, проверка
активного/неактивного пользователя или недостоверного отеля осуществляется по id пользователя/отеля,
однако методы через которые, теоретически можно обратиться к сответстующим сервисам или эндпоинтам
монолита в классе BookingServices реализованы.
- Вне зависимости от успешности бронирования сервис "booking" отправляет данные сервису "booking_history",
ответственному за сохранение записей истории бронирования.
- Помимо данных самого бронирования (если бронирование было успешным), в записи истории хранится флаг успешности
бронирования и причина отказа, в случае, если бронирование было неудачным. В реальном проекте, созданное бронирование
и запись должны быть разделены и связаны между собой связью "Один-к-Одному", но в учебной реализации я использовал одну
таблицу для экономии времени.
- При обращении к API бронирования монолита, монолит вызывает сервис "booking" используя протокл gRPC, сервис "booking"
осуществляет условную проверку полученых данных и в случае их валидности сохраняет данные в базу. Вне зависимости от
успешности бронирования в сервис истории бронирований отправляется информация о попытке бронирования с указанием причины
отказа или данными бронирования.
- Сервисы booking и booking_history используют Kafka для связи. Сервис "booking" не ожидает подтверждения или ответа от
сервиса booking_history, таким образом операци отправки данных по истории бронирования являются полностью асинхронными,
а сервмис "booking_history" имеет минимальную связанность с сервисом "booking".
