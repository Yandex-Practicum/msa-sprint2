## Стратегии миграции данных:
- Перенос данных в рамках одного перемещения  
    +: Простота реализации  
    -: Возможность потери данных

- Начать с blue-green deployment и направить часть трафика на микросервис (с помощью Load Balancing + ip hash) и хранить новые данные там и батчами мигрировать старые данные из монолита.  
    +: Позволяет сразу тестировать процесс перемещения данных, минимальный downtime  
    -: Сложность реализации, ip клиента может меняться, нужно дополнительно продумывать, в остальных сервисах монолита могут быть проблемы с консистентностью данных  
 
- Постепенно переносить данные и с помощью ACL получать данные которые есть в микросервисе из него, а неперенесенные записи получать из монолита. Микросервис в таком случае будет выполнять роль этакого кеша  
    +: Миграция незаметна для конечного пользователя, надежная миграция, минимальный downtime, мигрирует по ходу.  
    -: Сложная реализация, сложность обеспечения консистентности данных  

### To Be

Будет применена совокупность описанных подходов:

1) Будет внедрен ACL слой через который будут проходить все операции связанные с мигрируемыми данными
2) Новые данные будут записываться в микросервис и монолит
3) В фоне будет проходить миграция данных батчами
4) Полученные данных (на уровне ACL) сначала идет в микросервис, затем монолит
5) Трафик к микросервису направляется по схеме blue-green deployment, чтобы посмотреть на реакцию системы
6) После окончания миграции данных функционал монолита связанный с перемещаемыми данными выводится из эксплуатации.
